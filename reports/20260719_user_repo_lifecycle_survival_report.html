유저 생존의 핵심은
“이탈하지 않는 것”이 아니다

28일 무활동을 이탈로 정의하면 대부분의 유저는 결국 이탈한다. 중요한 것은 첫 이탈을 늦추고, 이탈한 유저가 다시 돌아올 수 있는 레포 활동 루프를 만드는 것이다.

2026-07-19 · GitHub Archive PushEvent 기준 개발자 · UTC · 2025-05-01~2026-07-18

1. 분석 결론 — 무엇이 중요한가

핵심 결론: 관찰상 개발자 생존과 가장 강하게 연결된 선행 신호는 멀티 repo나 협업보다 4주 동안 같은 repo로 반복 복귀한 기록이다. 여러 repo 활동과 공동 Push 경험은 그 다음의 추가 신호다. 이 관계가 원인인지는 실험으로 확인해야 한다.

1주기성이 1순위

과거 4주 중 1주만 Push한 활성 개발자의 다음 28일 이탈률은 52.2%, 4주 모두 Push하면 8.2%다.

2반복 repo가 핵심

2주 이상 Push한 repo가 없으면 이탈률 47.5%, 반복 repo가 2개 이상이면 7.9%다.

3이탈은 끝이 아님

완전한 180일 후속 관측이 가능한 첫 관측 이탈자 중 46.5%는 다시 Push한다. 이탈 방지와 복귀를 별도 문제로 관리해야 한다.

하지 말아야 할 해석: repo를 억지로 늘리거나 Push 횟수를 기계적으로 늘리면 수명이 길어진다는 뜻이 아니다. 관찰된 최우선 건강 신호는 “다음 주에도 실제 개발 작업을 이어갔는가”이며, 액션은 비교군이 있는 실험 후보일 뿐이다.

가능한 CRM 예시: 7일 동안 추가 Push가 없으면 최근 repo 이름과 함께 “지난 작업 이어보기” 알림을 한 번 보낸다. 15~21일 동안 Push가 없으면 마지막 repo나 열려 있는 작업을 다시 보여준다. 28일 이탈 후에는 “다시 시작하세요” 같은 일반 문구보다 마지막으로 활동한 repo를 중심으로 복귀 메시지를 보낸다. 이는 확정된 처방이 아니라 실험 가능한 예시이며, 알림을 받지 않은 비슷한 개발자와 다음 주 동일 repo Push율과 28일 복귀율을 비교해 효과를 확인해야 한다.

3. 근거 — 왜 이 액션 순서인가

2026-02-01~04-19의 주간 기준일 12개에서, 기준일 직전 7일에 Push한 활성 개발자만 비교했다. 이전 28일 행동과 이후 28일 이탈, 이탈 후 56일 내 복귀를 시간상 분리했다.

여러 repo 활동은 초기 보조 신호

같은 활동 주 수 안에서도 여러 repo에서 활동한 개발자의 이탈률이 낮았다.

  • 1주 활동: 1 repo 53.9% vs 2+ repo 41.7%
  • 2주 활동: 36.0% vs 31.5%
  • 3주 활동: 22.9% vs 18.9%
  • 4주 활동: 9.2% vs 7.8%

차이는 초기 집단에서 가장 크고, 이미 4주 내내 활동한 집단에서는 작다.

이 28/84/180일 지표는 완전한 각 후속 관측이 가능한 첫 관측 이탈 cohort 기준이며, 왼쪽 표의 기준일 표본·56일 복귀율과 분모가 다르다. 복귀자의 중앙 복귀 시점은 이탈 판정 후 43일이다.

Create·Delete 추가 분석: Create만 있었던 개발자는 같은 Push 활동 주와 repo 수를 맞추면 이탈률 차이가 거의 없었다. 반면 Delete가 있었거나 Create와 Delete가 모두 있었던 개발자는 대부분의 비교 구간에서 이탈률이 더 낮았다. 예를 들어 2주 활동 개발자의 이탈률은 둘 다 없을 때 35.0%, Delete만 있을 때 27.0%, Create와 Delete가 모두 있을 때 23.5%였다. Delete를 유도하자는 뜻이 아니라, 작업 정리 단계까지 간 개발자를 찾는 상태 신호로 사용한다.

6. 해석 시 주의할 점

  • user_id는 GitHub 이벤트를 만든 계정의 ID다. 명시적 bot과 기계 속도 계정은 제외했지만, 일반 이름을 쓰며 낮은 속도로 동작하는 자동화 계정은 남을 수 있다.
  • 사람 계정으로 AI 도구를 함께 사용한 개발은 포함한다. 이 데이터만으로 사람의 AI 보조 사용 여부를 구분할 수 없다.
  • 최초 관측은 가입일이 아니다. 데이터 시작 전 활동은 볼 수 없어 left-censoring이 있다.
  • 28/84/180일 복귀율은 각 기간을 끝까지 관측할 수 있는 이탈 집단만 포함한다.
  • 28일 이탈은 운영 정의다. 계정의 영구 이탈이나 GitHub 탈퇴를 의미하지 않는다.
  • 활동 주기, 멀티 repo, 협업과 장기 활동의 관계는 예측적 연관성이며 인과효과가 아니다.
