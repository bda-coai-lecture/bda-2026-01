6 · AI ANALYTICS PLAYBOOK

AI가 잘하느냐보다 먼저
데이터 계약이 명확한가를 묻는다

AI 분석의 성공과 실패를 문제의 난이도가 아니라 정의·분석 단위·시간·식별자·계측·검증 가능성으로 나눠 본다.

핵심 결론: AI가 잘하느냐 못하느냐는 “분석 난이도”보다 데이터 계약이 명확한가에 갈린다. 실무에서는 분석가를 대체하는 존재보다 검증 가능한 일을 빠르게 수행하는 보조 분석가이자 데이터 감사자로 두는 편이 정확하다.
용어 먼저: raw는 가공 전 원천 데이터, mart는 분석하기 좋게 정리한 표, grain은 한 행이 나타내는 단위다. 예를 들어 사용자별 표와 회사별 표는 grain이 다르다.

2. AI가 강했던 일

METRIC

정의 기반 지표 조회

“최근 30일 활성 사용자”가 로그인 기준인지 구매 기준인지 정의되어 있다면, SQL 작성과 월별 반복 집계를 안정적으로 수행한다. 데이터가 없을 때 없다고 답하는 것도 중요한 품질이다.

AUDIT

정합성 감사와 반례 추적

A/B 테스트의 사용자 배정표와 이벤트 로그를 비교해 한 사용자가 두 그룹에 들어간 경우를 찾고, 여러 실험에 공통된 배정 로직 문제인지 좁힐 수 있다.

INCIDENT

데이터 누락 원인 분해

원천 1,000건과 분석 테이블 980건이 다를 때 실제 유실, 필터 제외, 갱신 지연, 중복 제거를 따로 계산해 “20건 누락”을 여러 원인으로 분해한다.

SIMULATION

백테스트와 시뮬레이션

새 위험 기준을 과거 데이터에 적용해 승인·거절 대상이 얼마나 바뀌는지 계산하고, 기존 기준과 결과가 달라지는 사례를 목록으로 만든다.

TAXONOMY

대량 로그 의미 분류

수천 개 로그를 사용자 행동, 시스템 자동 작업, 오류로 분류해 단순 호출 수를 실제 사용량으로 오해하는 문제를 줄인다.

COMMON TRAIT

공통점은 검증 가능성

성공 사례는 모두 숫자와 정의를 다시 쿼리하거나, 다른 원천과 대조하거나, 실패 반례를 찾을 수 있었다.

3. AI가 약했던 일

취약점대표 사고왜 깨졌나필요한 가드레일
이름을 믿는 해석completed_at을 실제 결제 완료 시각으로 가정컬럼명과 실제 제품 의미가 다를 수 있다.코드·이벤트 발생 지점·제품 정의 확인
분석 단위 불일치사용자 단위 실험을 회사 수로 집계한 회사에 여러 사용자가 있어 분모가 바뀐다.입력·JOIN·출력 단위를 먼저 표기
식별자·시간대외부 ID로 조인하거나 UTC 날짜를 한국 날짜로 해석동일 대상을 놓치고 자정 부근 이벤트가 다른 날로 이동한다.대표 ID와 기준 시간 계약
계측 공백과 대리 지표메시지 발송과 방문을 연결할 키 없이 캠페인 효과 추정방문이 메시지 때문인지 직접 관측할 수 없다.계측 공백을 결론보다 먼저 보고
인과·정책·서사비교 집단 없는 기능 출시 후 매출 상승을 기능 효과로 표현계절성·광고·사용자 구성 변화와 구분할 수 없다.사람 승인, 비교 집단, 독자별 설명 리뷰

4. 좋은 결과는 모델보다 절차에서 나온다

실행 전 강제 조건

  • dbt 지표 정의 먼저 확인
  • 검증된 mart를 raw보다 우선
  • 사내·테스트·bot 계정 제외
  • 고정 날짜 대신 실행 기준일 사용
  • 확인하지 않은 대시보드 링크를 만들지 않음
  • 운영 공간이 아닌 임시 공간에서 검증

실패 기준을 미리 정한다

답변을 숫자·지표·이벤트·테이블·기간 주장으로 쪼개고 재쿼리, dbt 정의, 코드 검색으로 확인한다.

즉시 재검토: 잘못된 테이블/컬럼, JOIN 오류, 제외 조건 누락, 존재하지 않는 이벤트, 허용 범위를 넘는 수치 차이.

5. 운영 역할을 이렇게 나눈다

AI에게 맡기기 좋은 일

  • 정의 기반 지표 조회와 반복 SQL
  • raw-vs-mart, old-vs-new parity
  • 대시보드·카드 전수 점검
  • 로그 taxonomy 분류
  • 백테스트·시뮬레이션·case study
  • 장애 원인 후보를 넓히고 좁히기

사람이 앞서야 하는 일

  • 지표 정의 자체와 proxy 허용 판단
  • 제품·정책 의사결정
  • 고객·임원용 메시지
  • prod Slack 등 운영 커뮤니케이션
  • 수기 시트와 운영 DB 충돌 시 최종 정본 결정

6. 실전용 사전 점검

  1. 정의: 지표의 분자·분모·완료 조건이 있는가?
  2. grain: user·company·workspace·event 중 무엇인가?
  3. 시간: UTC/KST, event date/export date 중 무엇인가?
  4. 식별자: 같은 대상을 연결할 대표 ID와 1:1·1:N 관계가 정해졌는가?
  5. 제외: 자사·bot·test·중복·불완전 cohort 조건이 있는가?
  6. 신선도: 원천 갱신 지연, 변경 데이터 누락, 최신 날짜 데이터의 완결성을 확인했는가?
  7. 검증: raw·mart·code·log 중 두 개 이상으로 반례를 찾을 수 있는가?

앞의 여섯 조건 중 하나라도 비면 AI의 자신감보다 계약 보완을 먼저 한다. 검증 불가능한 질문은 “분석 실패”가 아니라 “계측·정의 작업이 먼저 필요한 질문”으로 돌려보낸다.
