주제: GitHub(GH Archive) DAU/MAU 하락 진단 · 2026-07-12 · 본 자료는 최종 보고서와 별개의 과정 기록입니다
💡 이 자료의 핵심 메시지: 정답을 처음부터 아는 게 아니라, 좋은 질문과 반박이 분석을 만든다. 사람이 방향을 잡고(“이 비교 이상하지 않아?”), 실행이 데이터로 답한다. 아래는 그 왕복(티키타카)의 기록입니다.

0. 출발점 — 무엇을 보고 있었나

GitHub 공개 활동 로그(GH Archive) 기반 지표에서 DAU/MAU가 눈에 띄게 하락. 초기 진단은 “유저 이탈이 아니라 데이터 수집 쪽 문제”. 이를 뒷받침할 진단용 마트(diag_*)와 Metabase 대시보드 진단 카드가 이미 구축된 상태에서 시작.

1. 대화가 분석을 바꾼 8개의 장면

각 행은 “사람이 던진 방향 → 왜 중요한 쟁점인지 → 실행이 한 대응 → 남은 교훈”.

# 방향 제시 (사람) 쟁점 · 왜 중요한가 대응 (실행 + 데이터) 교훈
1 “7월은 이제 막 시작인데, 4월이랑 그냥 비교해도 돼?”비교 방법론 의심 30일치(4월)와 11일치(7월) 총합을 그대로 비교하면 7월이 날짜가 덜 쌓였다는 이유만으로 낮게 나와 감소가 과장됨. MAU는 롤링 윈도우라 더 위험. 두 가지 정합 비교로 교체 — A: 완성월끼리(4월 vs 6월), B: 같은 창끼리(1–11일). 모든 지표를 일평균으로 정규화. 부분기간 통짜 비교 금지
2 “보고서 두괄식으로. 결론+수치 먼저, 비즈니스 언어로, 기술은 하단.”전달 방식 분석이 맞아도 읽는 사람이 결론을 못 잡으면 소용없음. 이해관계자는 “겉보기 하락 vs 실제 건강도”가 한눈에 필요. 맨 앞에 요약(3열 대조표) + 누구나 아는 레포로 만든 스토리를 상단에, 검증·쿼리는 기술 부록으로 완전 분리. 결론 먼저 · 청중 언어
3 “예시로 든 anthropics/skills는 최근 레포잖아?”예시 전제 점검 “꾸준히 인기 있던 레포가 무너졌다”는 논거인데, 최근 출시라 원래 감쇠 중이면 예시로 부적절. 수년째 안정적인 torvalds/linux 등으로 교체. 예시의 전제를 검증
4 “‘안 잡힌다’면 유실이란 거야? 원본이랑 교차검증하면 되잖아. 그냥 트렌드가 빨리 바뀐 거 아냐?”가장 날카로운 반박 기존 “원본 vs 우리 데이터 99.6% 일치”는 같은 혈통끼리의 대조라 순환논리 — 우리 충실도만 증명할 뿐, 원본의 진위는 증명 못 함. 완전히 독립된 소스(GitHub 라이브 스타 카운터)로 실제 속도 측정: build-your-own-x는 지금도 ~5,800/월인데 우리 데이터엔 ~2%만. 검증은 독립 소스로
5 “문서화 말고 구멍을 메워. 우리 데이터 파이프라인 이상 아냐?”원인 지점 추궁 실제 별은 나는데 우리엔 2%뿐 → 어디서 새는가를 층별로 특정해야 함(우리 탓이면 고쳐야 함). 3층 측정: 실제 GitHub(정상) → GH Archive 원본(글로벌 WatchEvent 하루 863건, 붕괴) → 우리 fact(원본을 충실 반영). 인제스트 코드에 이벤트 타입 필터 없음 확인. 누수를 층별로 특정
6 “BQ의 GH Archive가 source of truth야. 원본에 없으면 없는 거. 중간 파이프라인 검사만 빡세게 해.”스코프 교정 — 핵심 전환 외부 API로 원본을 감사하거나 우회 수집하는 건 분석 경계 밖 · 오버엔지니어링. 우리가 통제하는 유일한 지점은 원본→마트 전환. 라이브 API 우회 수집기·워치독·임시 테이블 전부 폐기. 대신 dbt 중간 검증 4종 추가 — 전부 통과(이벤트 수 보존 0건 위반). 경계 존중 · 통제점만 검증
7 “레포가 인기를 잃은 게 아니라, 몇 달 전 잠깐 늘었던 건 아냐?”대안 가설 · 반증 요구 절벽 직전(2–4월)이 일시적 스파이크였다면 “붕괴”가 아니라 “정상 복귀”일 수 있음. 풀 히스토리 확인: linux는 2025-09~2026-04 내내 월 900~1,650으로 평탄 → 5월부터 붕괴. 스파이크 가설 반증, 예시가 더 단단해짐. 반증 시도로 견고화
8 “웹서치로 공론화된 적 있는지 다각도로 봐줘.”외부 근거 같은 현상이 실제 세계에서 보고된 적 있는지 확인하면 진단의 설득력이 올라감. 동일 실패 모드 선례 확인(GitHub Community #178788: 2025-10 GH Archive 손실, WatchEvent −38% 등, Event API 아웃티지). 디폴트 분석은 유실 무전제로 두고, 이 해석은 별도 레이어(§T7)로 분리. 외부 방증 · 단 분리

2. 관통하는 교훈

① 기간을 맞춰라

부분월 총합 비교는 감소를 과장한다. 완성월·동일 창·일평균(rate)으로 정규화.

② 순환논리를 경계하라

같은 소스에서 파생된 두 데이터의 일치는 “진위”가 아니라 “충실도”만 증명한다. 검증은 독립 소스로.

③ 경계를 지켜라

정의된 원본이 source of truth. 원본에 없으면 없는 것. 밖으로 나가 감사·우회하는 건 과잉.

④ 통제점만 검증하라

못 고치는 상류 대신, 우리가 소유한 원본→마트 전환의 무결성을 기계적으로 보증(dbt).

⑤ 반증을 시도하라

“스파이크 아니냐”처럼 결론을 깰 가설을 직접 던져 통과시키면 예시가 단단해진다.

⑥ 청중을 먼저 생각하라

결론·수치를 앞에, 근거·쿼리는 부록에. 같은 분석도 전달 구조로 설득력이 갈린다.

4. 장기적 관점 — 일회성 진단에서 지속 가능한 관측으로

보고서는 한 시점의 스냅샷이다. 진짜 가치는 이 분석이 마트 → 대시보드 → 자동 검증으로 이어져, 다음에 또 이런 일이 생겼을 때 스스로 드러나는 데이터 환경으로 남는 데 있다. (“실패를 잡는 것”이 아니라 “실패가 보이는 환경”을 만드는 관점)

요지: 좋은 분석의 끝은 “결론 문서”가 아니라 같은 문제가 재발하면 저절로 보이고, 파이프라인이 스스로 무결성을 증명하는 상태다. 마트·대시보드·dbt 검증이 그 지속 장치다.

🎯 한 줄 결론: DAU 하락의 정체는 “코어 유저 이탈”이 아니라 “특정 이벤트 종류(소셜)의 기록 급감”. 우리 파이프라인은 무결(dbt로 보증), 원인은 상류 공개 이벤트 스트림 — 그리고 이 결론은 8번의 반박을 통과하며 다듬어졌다.
