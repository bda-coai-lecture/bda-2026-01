# GitHub(GH Archive) DAU/MAU 감소 진단 리포트

- **작성일**: 2026-07-12
- **데이터**: `bda-coai.mart` (GH Archive 기반 마트), 관측 구간 2025-09-01 ~ 2026-07-11

---

## 요약

> **결론: 2026-05부터 공개 소셜 활동(별·포크·PR·이슈) 기록이 90%+ 급감했고 DAU가 그만큼 하락했다.** 단 이 하락은 **라이트 소셜 유저에 집중**되며 핵심 커밋 활동·재방문율은 그대로다 — 즉 코어 유저층의 건강 문제가 아니라 특정 이벤트 종류의 기록이 급감한 데서 온다. 하락한 DAU를 코어 이탈로 오독하지 않는 것이 핵심이다.

> **읽는 법**: 본 리포트의 기본 분석(§T1~T6)은 **데이터를 액면 그대로 — 유실을 전제하지 않고** 다룬다(기록된 급감은 곧 우리 데이터의 상태다). 이와 별개로, 이 급감이 상류 GitHub 이벤트 API 손실이라는 **알려진 실패 모드와 일치한다는 외부 방증**을 §T7에 해석 레이어로 덧붙였다. 기본 결론은 그 해석에 의존하지 않는다.

**한눈에 보는 수치** (2026-04 → 2026-06, 완성월 일평균 기준)

| 겉으로 보이는 하락 | 코어 유저층은 그대로 | 하락의 정체 |
|---|---|---|
| DAU **−9.4%** (최신 7월 기준 −24%) | 총 활동량 **+3.0%**<br>핵심 개발 활동(커밋) **+19.3%**<br>재방문율(retention) **0.44 유지** | 별·포크·PR·이슈 등 "가벼운 활동" **−81%**<br>(7월엔 −93%) |

**무슨 일인가**

1. 우리 DAU는 여러 종류의 GitHub 행동(**커밋, 별(Star), 포크, PR, 이슈**)으로 유저를 센다.
2. 2026-05부터 원본(GH Archive)에서 **커밋을 뺀 나머지 가벼운 행동 기록이 급감**했다(원인은 상류 공개 이벤트 스트림).
3. 그 결과, **가벼운 행동으로만 집계에 잡히던 라이트 유저**(예: 한 달에 별 한 번 누르는 사람)가 DAU에서 빠졌다. 반면 커밋을 남기는 핵심 개발자는 그대로 잡히고, 오히려 늘었다 — 하락이 **어느 층에서 나오는지**가 분명하다.

**그래서 뭘 해야 하나** (상세 §"권고")

- **① 코어 유저 이탈 대응은 불필요.** 하락은 라이트 소셜 층에 국한되고 코어 활동·리텐션은 건강하다. 전사 리텐션 캠페인은 번지수가 틀렸다.
- **② 추천·트렌드·인기 지표는 지금 신뢰 불가.** 추천이 쓰는 신호의 97%가 급감한 가벼운 활동에서 나온다 → **2026-05 이후 데이터로 재학습 금지**.
- **③ 파이프라인 무결성 검증.** 원본→마트 전환에서 **우리가 이벤트를 떨구지 않음을 dbt 검증으로 상시 보증**한다(§T4).

## 눈으로 확인하는 스토리 — 인기 레포들의 기록된 스타가 일제히 무너졌다

숫자 대신 **누구나 아는 레포**로 보면 이게 개별 레포 인기 문제가 아니란 게 드러난다. 아래는 우리 데이터에 기록된 **월별 신규 스타**다 — 셋 다 수년간 꾸준히 별이 붙던 안정적 레포다.

| 레포 (우리 데이터 기록, 월별 신규 스타) | 2026-02 | 03 | 04 | 05 | 06 | 07 |
|---|---|---|---|---|---|---|
| **codecrafters-io/build-your-own-x** | 2,620 | 3,031 | 2,347 | 1,079 | 419 | **46** |
| **public-apis/public-apis** | 2,316 | 3,446 | 1,792 | 1,084 | 240 | **55** |
| **torvalds/linux** (리눅스 커널) | 1,194 | 1,309 | 930 | 385 | 129 | **21** |

핵심은 **어느 한 레포가 아니라 서로 무관한 레포들이 같은 달에 똑같은 절벽으로 동시에** 떨어졌다는 것이다. 리눅스 커널처럼 20~30년째 꾸준한 레포까지 3개월 만에 기록상 스타의 98%가 사라지는데, 그게 세 레포에서 **동시에·균일하게** 일어난다. 개별 레포의 인기 부침이라면 시점도 폭도 제각각이어야 하는데 그렇지 않다. 이건 특정 레포의 인기 문제가 아니라 **소셜 이벤트 기록 전반이 동시에 무너진** 시스템적 패턴이다. (이 패턴이 상류 이벤트 API 손실이라는 알려진 사례와 일치한다는 외부 방증은 §T7.)

**"최근 잠깐 튄 게 빠진 것 아니냐"는 아니다.** 더 긴 히스토리를 보면 이 절벽은 스파이크의 되돌림이 아니라 **안정 베이스라인의 붕괴**다. 리눅스는 2025-09~2026-04 8개월 내내 월 **900~1,650**개로 지극히 평탄했고(2·3·4월 1,194·1,309·930은 그 한복판), build-your-own-x는 같은 기간 월 3,400~8,500, public-apis는 월 2,100~9,000을 유지했다. 셋 다 1년째 유지되던 수준에서 5월부터 무너진 것이다.

> 대조 — 진짜 개별 인기 부침은 이렇게 생겼다: `karpathy/autoresearch`는 2026-03 출시 직후 12,111 스타로 터졌다가 4월 2,737로 자연 냉각됐다. 완만하고, 이 레포만의 시점이다. 반면 5월 이후의 붕괴는 무관한 레포들이 **동시에·균일하게** 무너진다 — "개별 인기 변화"와 "기록 자체의 소멸"은 형태가 다르다.

### 지표가 서로 반대로 움직인다

코어 유저층이 빠지는 이탈이라면 활동량·재방문율도 같이 빠져야 한다. 그런데 우리 데이터에선:

- **유저 수(DAU)는 −9%~−24%로 하락** 하는데,
- **총 활동량은 +3%, 커밋 활동은 +19%~+31%로 오히려 성장** 하고,
- **재방문율은 0.44로 4월 이후 그대로**다.

남은 층의 활동이 늘고 붙잡는 힘이 유지되므로, 하락은 **코어 유저층의 이탈이 아니라 다른 층에서 나온다.** 그 층이 누구인지는 아래에서.

### 하락은 어느 층에서 나오나

빠진 층은 **한 달에 하루, 별 하나 누르고 마는 라이트 유저**다. 이들은 월 평균 활동일 1.3일, 이벤트 3.6개에 그치고 **82%가 그 달에 하루만 활동**한다 (핵심 개발자는 월 20개·3.8일). 봇도 아니다 — 봇은 전체의 1% 미만이고, 사람 개발자 활동은 오히려 늘었다.

이들은 별·포크·이슈 같은 가벼운 행동으로만 우리 데이터에 흔적을 남긴다. 그 행동이 원본에서 사라지면 **이 층이 통째로 DAU에서 빠진다** — 반면 커밋을 남기는 핵심 개발자는 그대로 집계된다.

## 권고 (Action Items)

**A. 지표 해석 (긴급)**
- 당분간 **커밋(Push) 기반 핵심 개발자 활동**을 1차 KPI로 분리해 본다. 전체 DAU는 원본의 소셜 커버리지 급감을 그대로 반영하므로, 코어 유저 건강도 판단에는 부적합하다.
- 대시보드에 하락의 원인 주석(소셜 이벤트 커버리지 급감)을 달아 이탈로 오독되지 않게 한다.

**B. 파이프라인 무결성 검증 (핵심)**
- **원본→마트 전환에서 이벤트를 떨구지 않음**을 dbt 테스트로 상시 보증한다: 원본 대비 fact **이벤트 수 보존**, grain(user·repo·action·date) **유일성**, event_count ≥ 1, activity_date 상식 범위. (구현: §T4)
- 원본 자체의 볼륨/분포 감시는 별도 신설 대신 **기존 dbt validate 체계**에 흡수한다(중복 방지).

**C. 커뮤니케이션**
- 이해관계자에게: "**하락은 라이트 소셜 유저 층에 국한되고, 코어 개발 활동·재방문율(0.44)은 건강하다.** 소셜 활동 기록 급감의 원인은 상류(공개 이벤트 스트림)에 있다." 전사 이탈 대응은 번지수가 틀렸다.

**D. 추천 시스템(ghrec) 보호 (긴급)**
- **2026-05 이후 데이터로 재학습 금지.** 추천이 쓰는 상호작용 신호의 97%가 급감한 가벼운 활동이라, 재학습하면 개인화가 무너진다.
- **커버리지가 정상이던 2026-03 이전 모델/데이터를 고정**해서 쓴다.

## T1. 비교 방법론 — 왜 이 숫자들인가

오늘은 7월 12일이라 7월은 11일치 부분월이다. 30일치 4월과 11일치 7월의 **총합(totals)을 그대로 붙이면** 7월이 날짜가 덜 쌓였다는 이유만으로 낮게 나와 감소가 과장된다. 그래서 두 가지 정합 비교로 본다.

- **A(메인) — 완성월끼리**: 2026-04 전체 vs 2026-06 전체(마지막 완성월, 둘 다 30일).
- **B(보조) — 같은 창끼리**: 2026-04 1–11일 vs 2026-07 1–11일(둘 다 11일). 최신성을 보되 날짜 수를 맞춤.

모든 유저·활동 지표는 **일평균(rate)** 으로 정규화해 부분월 편향을 제거했다.

**A. 완성월 비교 — 2026-04 vs 2026-06 (둘 다 30일)**

## T3. 검증: "사라진 유저 = 라이트 사람 유저"

### (a) 순 감소분의 대부분이 "소셜만 하고 Push는 안 하는" 유저

| 월 | Push 유저 | 소셜-온리 유저 | 전체 MAU |
|---|---|---|---|
| 2026-01 | 3,932,893 | 1,184,316 | 5,802,953 |
| 2026-03 | 4,946,276 | 933,089 | 6,700,971 |
| 2026-04 | 5,009,608 | 872,535 | 6,703,768 |
| 2026-05 | 5,134,707 | 548,416 | 6,500,326 |
| 2026-06 | 4,950,815 | **217,209** | 5,982,318 |

소셜-온리 유저 87.3만(4월) → 21.7만(6월), **−75%**. Push 유저는 유지. 4→6월 MAU 순감소 ~72만 중 소셜-온리 감소가 ~65만 → **순 감소의 대부분이 소셜-온리 유저.**

### (b) 그 소셜-온리 유저는 실제로 라이트 유저 (2026-04 기준)

| 코호트 | 유저 수 | 평균 활동일 | 평균 이벤트 | one-day 유저 비율 |
|---|---|---|---|---|
| push_user | 5,009,608 | 3.77일 | 20.53 | 35.8% |
| social_only(non-push) | 1,694,160 | **1.34일** | **3.61** | **82.2%** |

### (c) 봇이 아니라 사람 — 코어 활동은 성장 중

| 표본일 | 사람 Push 유저 | 사람 소셜/기타 유저 | 봇 유저(전체) |

| 2026-03-15 | 472,688 | 272,786 | ~4,435 |

봇은 활성 유저의 1% 미만. "Push 증가가 봇 때문"은 **틀렸다** — 사람 Push 유저가 47만 → 61만으로 증가.

| 2026-04 | 8,688,172 | 621,468 | 11,486,891 |
| 2026-06 | 9,174,637 | **176,279** | 11,753,398 |

소셜만 받는 레포 109만(1월) → 17.6만(6월), −84%. Push 레포는 오히려 증가. 즉 사라진 건 "코드가 커밋되는 레포(생산)"가 아니라 **"별·포크·이슈로만 소비되던 레포(발견/인기)"** → GitHub의 **생산 그래프는 온전히 보지만, 소비·인기·소셜 그래프에는 사실상 눈이 멀었다.**

## T4. 중간 파이프라인 검증 — 급감이 우리 전환 로직 탓이 아님을 보증

소셜 이벤트 급감이 원본이 아니라 우리 원본→마트 전환 로직 탓일 가능성을 배제해야 한다. 코드 확인과 상시 테스트로 배제했다.

**코드 확인 (dbt).** `raw_githubarchive_events_90d`(뷰) → `fact_user_repo_activity`(incremental)는 `githubarchive.day.*`에서 **이벤트 타입 필터 없이** 읽고 `COUNT(*)`로 집계만 한다(`group by user_id, repo_id, action, activity_date`, NULL 방어만). 소셜 이벤트를 떨구는 로직 자체가 없다. Push가 원본보다 적어 보이는 건 grain 압축(하루 여러 push→1행, event_count 보존)일 뿐 손실이 아니다.

**추가한 dbt 검증 (상시).** 코드 리뷰만으로 끝내지 않고, 매 빌드마다 원본↔fact를 기계적으로 대조하는 테스트를 넣었다 (`dbt/gharchive_metrics/tests/`, 현재 전부 통과):

| 테스트 | 무엇을 보증하나 |
|---|---|
| `assert_fact_event_count_conservation` | 빌드 윈도우 내 **날짜별 `SUM(event_count)` == 원본 이벤트 수** — 전환에서 이벤트를 하나도 안 흘렸는지 (핵심). 현재 **0 violations** |
| `assert_fact_grain_unique` | grain `(user_id, repo_id, action, activity_date)` **유일성** — incremental insert_overwrite가 파티션을 중복 적재하지 않았는지 |
| `assert_fact_event_count_positive` | `event_count ≥ 1` — 집계 산출물의 위생 |
| `assert_fact_activity_date_sane` | `activity_date`가 **상식 범위(2025-01-01 ~ 오늘)** 안 — suffix/파싱 버그로 엉뚱한 파티션(예: 2002년, 미래)이 새지 않았는지 |

> **결론**: 보존 테스트가 통과하므로 fact는 원본을 손실 없이 반영한다. 즉 소셜 이벤트 급감은 우리 전환 로직이 만든 것이 아니라 **원본에 이미 반영된 것**이다.

## T5. 추천 시스템(ghrec) 오염 상세 — 위험도 높음

ghrec의 implicit-feedback 가중치(`WEIGHTS`, 전 노트북 공통):

```
PullRequest 3.0 · Fork 2.0 · Watch 1.0 · Issue 0.5 · IssueComment 0.3   →  소셜 합 6.8
Push 0.2                                                                →  기술 신호 0.2
```

**추천 상호작용 신호의 약 97%(6.8/7.0)가 지금 붕괴한 소셜 이벤트.** 이 신호가 인기도 baseline → user-item 행렬 → ALS 임베딩 → LGBM 랭킹 → 서빙까지 전 단계에 흐른다(`src/ghrec/recommend.py`, `notebooks/ghrec/05·07·09`, `app_reco.py`). 2026-05 이후 데이터로 재학습하면 협업 신호가 사라져 개인화 붕괴 → **2026-03 이전 정상 구간 고정 사용.**

## T6. 재현 쿼리 (BigQuery, `bda-coai`)

```sql
-- 정합 비교: A(완성월 4월vs6월) + B(동일창 4월vs7월 1-11일), 모두 일평균
WITH d AS (
  SELECT activity_date, active_users, total_events,
    watch_events+fork_events+pull_request_events+issue_events+issue_comment_events AS social_events,
    push_events,
    DATE_TRUNC(activity_date, MONTH) AS month, EXTRACT(DAY FROM activity_date) AS dom
  FROM `bda-coai.mart.metrics_daily`)
SELECT period, COUNT(*) days,
  ROUND(AVG(active_users)) dau_avg, ROUND(AVG(total_events)) events_per_day,
  ROUND(AVG(social_events)) social_per_day, ROUND(AVG(push_events)) push_per_day,
  ROUND(100*SUM(social_events)/SUM(total_events),1) social_pct
FROM (
  SELECT 'A.4월전체' period, d.* FROM d WHERE month='2026-04-01'
  UNION ALL SELECT 'A.6월전체', d.* FROM d WHERE month='2026-06-01'
  UNION ALL SELECT 'B.4월1-11', d.* FROM d WHERE month='2026-04-01' AND dom<=11
  UNION ALL SELECT 'B.7월1-11', d.* FROM d WHERE month='2026-07-01' AND dom<=11)
GROUP BY period ORDER BY period;

-- 원천 vs 우리 fact 정합성 대조 (수집 버그 여부)
SELECT type, COUNT(*) FROM `githubarchive.day.20260615`
WHERE type IN ('WatchEvent','ForkEvent','PullRequestEvent') GROUP BY type;
SELECT action, COUNT(*) FROM `bda-coai.mart.fact_user_repo_activity`
WHERE activity_date='2026-06-15'
  AND action IN ('WatchEvent','ForkEvent','PullRequestEvent') GROUP BY action;

-- retention은 안정적임을 확인
SELECT week_start, active_users, retention_rate, churn_rate
FROM `bda-coai.mart.metrics_user_lifecycle_weekly`
ORDER BY week_start DESC LIMIT 12;
```

## T7. 외부 맥락 — 동일 패턴의 공개 선례 (해석 레이어)

> **이 절은 §T1~T6 기본 분석과 별개의 해석이다.** 기본 분석은 데이터를 액면 그대로(유실 무전제) 다루며 이 절에 의존하지 않는다. 다만 외부 검색 결과, 우리가 관측한 급감이 **상류 GitHub 이벤트 API 손실**이라는 이미 공론화된 실패 모드와 형태가 정확히 일치한다.

**선례: GitHub Community Discussion #178788 — "Significant data loss in GHArchive and OSSInsight"** (2025-11)
- **2025-10-08~14** GH Archive에 대규모 이벤트 손실: 월 −20%(89.7M vs 평소 112.2M), 피크일 −99%, 이후 일 5~10% 지속.
- 타격 이벤트 타입: PublicEvent −65%, CreateEvent −40%, **WatchEvent −38%, ForkEvent −37%**, ReleaseEvent −37% — **우리가 본 소셜 이벤트와 동일 집합**.
- 원인 판정(원문): **"GitHub Event API 아웃티지가 downstream으로 전파된 것, 파싱 버그 아님."** GH Archive와 **OSSInsight(Event API 직접 사용)에서 동시 관측** → 상류 API가 원인임을 교차확인.

**방증: GitHub Changelog (2025-08-08) — "Upcoming changes to GitHub Events API payloads"**
- 이벤트 API payload 축소(PR/Push 필드 제거, author_association 삭제). 브라운아웃 2025-09-08, 정식 적용 2025-10-07 — 위 10월 손실과 시점이 겹친다. GitHub이 이벤트 firehose를 실제로 손대고 있었다는 정황.

**정직한 한계.** 공개로 확정된 사건은 **2025-10**이고 규모도 일시적 −20% 수준이다. 우리 데이터의 **2026-05~ 90%+ 지속 붕괴**를 콕 집은 공개 스레드는 검색에서 확인되지 않는다. (a) 10월 사건이 악화·상시화된 미보고 연장선일 수도, (b) 본 데이터셋 특유의 상황일 수도 있다 — 단정하지 않는다.

**함의.** "우리 파이프라인 무관, 원인은 상류"라는 §T4 판정에 **외부 방증**이 붙는다. 동일한 실패 모드(상류 Event API → GH Archive 소셜 이벤트 손실, 동일 타입 집합, downstream 여러 도구 동시 타격)가 실제로 발생·공론화된 바 있다. 단, 기본 결론(라이트 소셜 층 집중·코어 건강·재학습 금지)은 이 해석 없이도 성립한다.
