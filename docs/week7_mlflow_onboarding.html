1. 왜 MLflow가 필요한가

추천 실험은 데이터 기간, candidate 생성, re-ranker, 평가 지표가 같이 움직인다. 기록 없이 여러 번 돌리면 “무엇을 바꿔서 좋아졌는지”를 설명하기 어렵다.

기록 없는 실험과 MLflow 실험

Before 노트북, CSV, 모델 파일이 흩어지고 run 조건을 나중에 복원하기 어렵다.
→
MLflow Run params, metrics, artifacts, git 정보를 한 실행 단위로 묶는다.
→
After 같은 기준으로 run을 비교하고, best run의 근거 파일을 바로 열 수 있다.
핵심 문제 모델을 잘 만드는 것만으로는 부족하다. 어떤 조건에서 어떤 결과가 나왔는지 남기지 않으면, 개선인지 우연인지 판단할 수 없다.

2. MLflow란 무엇인가

MLflow는 머신러닝 실험의 실행 기록을 남기는 시스템이다. 한 번 모델을 학습할 때마다 사용한 설정값, 나온 점수, 생성된 파일을 한 곳에 저장한다. 나중에는 웹 UI에서 여러 실행을 나란히 비교할 수 있다.

직관적 이해

MLflow는 실험의 조건, 결과, 산출물을 한 run 단위로 묶어두는 기록 시스템이다. 나중에 “왜 이 run이 좋았는지”를 다시 열어볼 수 있게 한다.

실험 관리 관점

단순한 실행 로그가 아니라 가설, 변경점, metric, artifact를 연결한다. 그래서 여러 run을 같은 기준으로 비교할 수 있다.

MLflow가 해주는 일

기록 대상 예시 왜 중요한가
Parameter candidate_k=300, lgbm_num_leaves=63 사람이 정한 실험 조건이다. 같은 조건으로 다시 실행하려면 필요하다.
Metric NDCG@10, Recall@100, elapsed_min 실행 결과다. 모델이 좋아졌는지 판단하는 기준이다.
Artifact metrics CSV, summary JSON, 모델 파일, diagnostics parquet 점수만으로 부족할 때 결과 파일을 열어 원인을 분석한다.
Tag script, git_sha, primary_model run을 분류하고, 어떤 코드 버전에서 나온 결과인지 추적한다.

3. MLflow 기본 개념

핵심 구조는 단순하다. Experiment 안에 여러 Run이 있고, 각 Run에는 Params, Metrics, Artifacts가 붙는다.

MLflow object model

Experiment 같은 목적의 run 묶음
Run 한 번의 실행 기록
Params 실행 전 정한 값
Metrics 실행 후 나온 점수
Artifacts 실행 결과 파일
Tags 코드/모델 분류 정보
Model files LGBM, PyTorch checkpoint
Diagnostics CSV, parquet, JSON

Experiment

Experiment는 실험을 담는 폴더 같은 개념이다. 예를 들어 이 저장소에서는 two-stage 추천 실험과 neural re-rank 비교 실험을 서로 다른 experiment로 나눈다.

  • recsys-two-stage: ALS candidate 생성과 LGBM re-ranker 중심 실험
  • recsys-rerank: LGBM, FM, Deep&Wide, DeepFM, DLRM re-rank 비교 실험

Run

Run은 한 번의 실행이다. 데이터 기간, candidate 개수, 모델 파라미터가 다르면 다른 run으로 남긴다.

Parameter와 Metric의 차이

Parameter는 실행 전에 사람이 정한 값이고, Metric은 실행 후 계산된 값이다. 이 둘을 구분하는 습관이 중요하다.

구분 질문 추천 실험 예시
Parameter 내가 무엇을 바꿨나? related_candidate_cap=80, factors=96
Metric 그 결과 무엇이 달라졌나? core_ndcg_at_10=0.016, core_recall_at_100=0.074

Artifact

Artifact는 실험의 증거 파일이다. 숫자만으로 부족할 때 원인을 확인하는 입구다.

  • metrics.csv: 모델별 NDCG, Recall, Precision, coverage
  • summary.json: 데이터 규모, split, feature, 경로 정보
  • week6_user_diagnostics_*.parquet: 유저별 hit, NDCG, source mix
  • diagnostics/*_by_*.csv, week6_segment_summary_*.csv: segment별 평균 성능
  • lgbm_ranker_*.txt, *.pt: 저장된 모델

4. Harness 관점으로 보는 MLflow

MLflow는 실험 전체를 대신 실행하는 도구가 아니다. 우리가 만든 실험 harness가 남기는 기록을 표준화해서 보여주는 관측 장치다.

추천 실험 harness architecture

Data Split history / rank-label / test
Feature Builder user, repo, candidate features
Retriever ALS, recent, popular, related
Re-rank LGBM, FM, DeepFM, DLRM
Evaluator NDCG, Recall, coverage
Diagnostics user segment, source mix
MLflow Tracking params, metrics, artifacts
MLflow UI compare, filter, inspect

무엇을 MLflow에 남기나

역할 이 저장소의 예시 MLflow와의 관계
Harness 실행기 scripts/week6_two_stage_v2.py 실험을 실제로 실행한다.
데이터 기준 history / rank-label / test split exp_data_* param으로 기록한다.
실험 스위치 candidate cap, related source, re-rank params exp_candidate_*, exp_ranker_* param으로 기록한다.
검증 기준 NDCG@K, Recall@K, unique@K metric으로 기록한다.
진단 자료 user diagnostics, segment summary artifact로 기록하고 필요할 때 파일을 열어 확인한다.

비교가 공정해지는 기준

candidate set이 달라진 상태에서 LGBM과 DeepFM을 비교하면 re-rank 모델 차이인지 candidate 품질 차이인지 알 수 없다. 그래서 split, candidate cache, metric 기준을 run마다 남긴다.

5. 추천시스템 실험에서 MLflow가 특히 중요한 이유

추천은 “모델 하나”가 아니라 candidate 생성과 re-rank의 조합이다. 그래서 run 기록도 pipeline 기준으로 남겨야 한다.

추천 실험은 보통 두 단계다

Two-stage recommender experiment

Retrieval ALS + recent + popular + related source로 candidate pool을 만든다.
→
Re-rank LGBM, DeepFM, DLRM이 candidate 순서를 다시 매긴다.
→
Evaluation NDCG@K, Recall@K, unique@K를 비교하고 segment 진단은 artifact에서 본다.

추천 실험에서 반드시 남겨야 하는 것

범주 기록할 값 MLflow key 예시
데이터 기간 history, rank-label, test 기간 exp_data_history_start, exp_data_test_end
candidate 생성 candidate K, hybrid extra, related cap exp_candidate_k, exp_candidate_related_cap
re-rank 설정 모델 종류, LGBM leaf, epoch, batch size exp_ranker_names, exp_ranker_lgbm_num_leaves
주요 성능 상단 품질과 candidate 회수 core_ndcg_at_10, core_recall_at_100
진단 성능 push, PR, high-activity 유저군의 성능 diagnostics/*_by_*.csv artifact
주의할 점 추천에서는 평균 NDCG 하나만 보면 위험하다. 전체 평균이 좋아져도 특정 유저군에서 나빠질 수 있고, 인기 repo 몇 개에만 추천이 몰리면 coverage가 나빠질 수 있다.

6. 이 저장소에서 MLflow 보기

이 저장소는 MLflow tracking URI로 sqlite:///mlflow.db를 사용한다. 즉 run 기록은 repo root의 mlflow.db에 저장된다.

MLflow UI 실행

cd bda-2
uv run mlflow ui --backend-store-uri sqlite:///mlflow.db --port 5000

실행 후 브라우저에서 http://127.0.0.1:5000을 열면 된다.

주요 experiment

Experiment 용도 주로 볼 것
recsys-two-stage ALS retrieval + LGBM re-rank 실험 candidate source, related cap, segment diagnostics
recsys-rerank LGBM/FM/Deep&Wide/DeepFM/DLRM re-rank 비교 같은 candidate set에서 re-rank 모델별 NDCG/Recall 비교
recsys-retrieval-legacy Two-Tower retrieval 비교 core_*는 Two-Tower 기준이다. ALS baseline은 모델별 metric이나 artifact에서 본다.

실험 스크립트

# two-stage 추천 실험
uv run python scripts/week6_two_stage_v2.py \
  --smoke \
  --save-user-diagnostics \
  --output-suffix mlflow_onboarding_smoke

# re-rank 비교 실험
uv run python scripts/week6_neural_rankers.py \
  --smoke \
  --rankers lgbm \

두 스크립트 모두 기본적으로 MLflow에 기록한다. 기록을 끄고 싶을 때만 --no-mlflow를 붙인다.

기존 산출물 재적재

uv run python scripts/backfill_mlflow_from_week6_artifacts.py

이미 문서에 정리된 과거 실험 결과는 재학습하지 않고 metrics CSV와 summary JSON에서 MLflow run으로 다시 적재할 수 있다. 이 명령은 two-stage candidate 실험, re-rank 비교, Two-Tower retrieval 결과를 판단 가능한 run table로 복원한다.

현재 repo의 MLflow 설계

MLflow run table은 강의 판단에 필요한 컬럼만 남긴다. 전체 args와 상세 모델별 metric은 artifact의 summary JSON과 metrics CSV에서 확인한다.

추천 실험에서 대표로 볼 key는 다음과 같다.

exp_candidate_* exp_data_sample_ratio exp_ranker_* (re-rank) core_ndcg_at_* core_recall_at_100 artifact diagnostics

7. MLflow UI 읽는 법

MLflow UI를 처음 열면 run 목록이 표로 보인다. 이 표를 “실험 비교표”로 읽으면 된다.

Experiment 목록에서 먼저 확인할 것 Description과 Tags의 experiment_role을 본다. candidate는 candidate source 실험, re-rank는 같은 candidate set 위의 re-rank 모델 비교다.
지표가 많아 보일 때 UI 첫 화면에서는 core_* metric만 본다. ALS, Popularity, 모델별 상세 metric과 segment 진단은 artifact의 metrics.csv, diagnostics/*_by_*.csv에서 확인한다.

처음 5분 순서

1. Experiment candidate 실험인지 re-rank 비교인지 먼저 고른다.
2. Status FINISHED run만 비교 대상으로 둔다.
3. Columns candidate 설정과 core_* metric만 켠다.
4. Sort core_recall_at_100 또는 core_ndcg_at_100으로 정렬한다.
5. Artifacts best run의 summary와 segment CSV를 연다.

첫 화면 column

목적 Column 읽는 법
데이터 규모 params.exp_data_sample_ratio, metrics.eval_users screening run과 full run을 섞어 비교하지 않는다.
candidate 생성 비교 params.exp_candidate_k, params.exp_candidate_related_cap Recall@100 변화가 candidate pool 변화 때문인지 확인한다.
관련 후보 설정 params.exp_candidate_related_top_per_anchor, params.exp_candidate_related_max_seen_anchors related source를 넣은 run끼리 세부 설정을 맞춘다.
품질 판단 metrics.core_ndcg_at_10, metrics.core_ndcg_at_100 상단 품질과 전체 ranking 품질을 같이 본다.
회수와 다양성 metrics.core_recall_at_100, metrics.core_unique_at_100 정답 후보 회수와 추천 쏠림을 같이 본다.
비용 metrics.elapsed_min 성능이 좋아도 너무 느리거나 추천이 쏠리면 보류한다.
첫 화면에서 할 일 가장 먼저 best score를 찾는 것이 아니라, 비교 가능한 run끼리 묶는 것이 먼저다. 데이터 기간과 candidate cache가 다르면 점수를 바로 비교하지 않는다.

비교 질문을 명확히 하기

질문 고정할 것 바꿀 것 볼 metric
related source가 도움이 됐나? split, ALS/LGBM 설정 exp_candidate_related_cap core_ndcg_at_10, core_recall_at_100, coverage
LGBM 튜닝이 도움이 됐나? candidate set LGBM params core_ndcg_at_10, core_ndcg_at_100
neural re-rank가 LGBM보다 좋은가? candidate cache, split exp_ranker_names core_ndcg_at_100, 모델별 상세는 metrics artifact
어떤 유저군이 약한가? best run segment diagnostics/*_by_*.csv, user diagnostics artifact

Run detail 보기

특정 run을 클릭하면 params, metrics, artifacts가 따로 보인다. 여기서 artifact를 열어 summary JSON, metrics CSV, diagnostics CSV를 확인한다.

8. 추천 실험 workflow

좋은 실험은 비교 질문을 정하고, 그 질문에 필요한 부분만 바꾸는 방식으로 진행한다.

1. Baseline기준 run 선택
2. Question비교 질문 작성
3. Change한 축만 변경
4. TrackMLflow run 기록
5. Decidemetric + artifact 확인

예시 1. Candidate source 실험

uv run python scripts/week6_two_stage_v2.py \
  --best-full \
  --related-candidate-cap 80 \
  --related-top-per-anchor 10 \
  --related-max-seen-anchors 20 \
  --save-user-diagnostics \
  --output-suffix related80_anchor20_full

이 실험의 질문은 “같은 유저가 함께 본 repo를 related source로 추가하면 candidate 회수와 상단 품질이 좋아지는가?”다. 따라서 MLflow에서는 related cap과 Recall@100, NDCG@10, segment summary artifact를 함께 본다.

예시 2. Re-rank 비교 실험

uv run python scripts/week6_neural_rankers.py \
  --reuse-feature-cache \
  --feature-cache-path data/models/week6/week6_ranker_compare_week7_full_500k_u100k_e3_features.pkl \
  --rankers lgbm,fm,deepwide,deepfm,dlrm \
  --epochs 3 \
  --batch-size 32768 \
  --torch-threads 1 \
  --device cpu \
  --output-suffix week7_ranker_compare

이 실험의 질문은 “같은 candidate set에서 어떤 re-rank 모델이 가장 좋은가?”다. candidate cache를 재사용해야 re-rank 비교가 공정해진다.

좋은 run 이름

  • 좋음: related80_anchor20_full_als96_i12_lgbm63
  • 좋음: week7_full_500k_u100k_e3_reuse_b32768
  • 나쁨: test, final, new_new_final

9. 실험 전후 체크리스트

실험 전

  • 질문이 한 문장인가?
  • baseline run이 있는가?
  • split과 candidate cache가 맞는가?
  • run 이름이 설명적인가?

실험 후

  • run status가 FINISHED인가?
  • NDCG, Recall, unique를 같이 봤는가?
  • segment summary artifact에서 약해진 유저군이 없는가?
  • artifact가 남았는가?
좋은 실험 해석 “NDCG가 올랐다”에서 멈추지 않고, 어떤 candidate source를 추가했는지, Recall@100과 coverage가 함께 좋아졌는지, 특정 segment에서 손실이 생기지 않았는지까지 확인한다.
