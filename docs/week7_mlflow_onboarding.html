1. 왜 MLflow가 필요한가

머신러닝 수업에서 처음에는 보통 노트북 하나로 시작한다. 데이터를 읽고, 전처리하고, 모델을 학습하고, 마지막 셀에서 점수를 확인한다. 작은 실습에서는 이 방식이 빠르다. 하지만 실험이 늘어나면 바로 문제가 생긴다.

  • 어제 나온 점수가 어떤 데이터 기간으로 만든 결과였는지 기억나지 않는다.
  • 후보 생성 설정을 바꾼 것인지, 랭커 설정을 바꾼 것인지 헷갈린다.
  • 노트북 셀을 순서대로 실행하지 않아 결과가 재현되지 않는다.
  • 최고 성능이라고 생각한 run의 모델 파일, feature importance, 평가 CSV가 흩어진다.
  • 평균 점수는 올랐는데 어떤 유저군에서 나빠졌는지 확인하기 어렵다.
핵심 문제 모델을 잘 만드는 것만으로는 부족하다. 어떤 조건에서 어떤 결과가 나왔는지 남기지 않으면, 개선인지 우연인지 판단할 수 없다.

MLflow는 이 문제를 풀기 위한 도구다. 모델 자체를 대신 만들어주는 도구라기보다, 실험을 기록하고 비교하고 재현할 수 있게 해주는 실험 관리 도구로 이해하면 된다.

2. MLflow란 무엇인가

MLflow는 머신러닝 실험의 실행 기록을 남기는 시스템이다. 한 번 모델을 학습할 때마다 사용한 설정값, 나온 점수, 생성된 파일을 한 곳에 저장한다. 나중에는 웹 UI에서 여러 실행을 나란히 비교할 수 있다.

비전공자식 설명

MLflow는 머신러닝 실험의 실험노트다. 요리로 치면 레시피, 재료, 조리 시간, 완성 사진, 맛 평가를 한 장에 붙여두는 기록장에 가깝다.

취준생식 설명

MLflow는 포트폴리오에서 “모델을 돌려봤다”를 “어떤 가설을 어떤 설정으로 검증했고, 어떤 metric과 artifact로 판단했다”로 바꿔주는 도구다.

MLflow가 해주는 일

기록 대상 예시 왜 중요한가
Parameter candidate_k=300, lgbm_num_leaves=63 사람이 정한 실험 조건이다. 같은 조건으로 다시 실행하려면 필요하다.
Metric NDCG@10, Recall@100, elapsed_min 실행 결과다. 모델이 좋아졌는지 판단하는 기준이다.
Artifact metrics CSV, summary JSON, 모델 파일, diagnostics parquet 점수만으로 부족할 때 결과 파일을 열어 원인을 분석한다.
Tag script, git_sha, primary_model run을 분류하고, 어떤 코드 버전에서 나온 결과인지 추적한다.

3. MLflow 기본 개념

MLflow를 처음 볼 때는 용어가 낯설 수 있다. 하지만 핵심은 단순하다. Experiment 안에 여러 Run이 있고, 각 Run에는 Params, Metrics, Artifacts가 붙는다.

Experiment 같은 목적의 실험 묶음
Run 한 번의 실행 기록
Params 실행 전 정한 설정
Metrics 실행 후 나온 점수
Artifacts 실행 결과 파일

Experiment

Experiment는 실험을 담는 폴더 같은 개념이다. 예를 들어 이 저장소에서는 two-stage 추천 실험과 neural ranker 비교 실험을 서로 다른 experiment로 나눈다.

  • bda-week6-two-stage: ALS 후보 생성과 LGBM 랭커 중심 실험
  • bda-week7-recsys-rankers: LGBM, FM, Deep&Wide, DeepFM, DLRM 비교 실험

Run

Run은 한 번의 실행이다. 같은 스크립트를 실행하더라도 데이터 기간, 후보 개수, 모델 파라미터가 다르면 다른 run으로 남긴다.

Parameter와 Metric의 차이

Parameter는 실행 전에 사람이 정한 값이고, Metric은 실행 후 계산된 값이다. 이 둘을 구분하는 습관이 중요하다.

구분 질문 추천 실험 예시
Parameter 내가 무엇을 바꿨나? related_candidate_cap=80, factors=96
Metric 그 결과 무엇이 달라졌나? primary_ndcg_at_10=0.016, primary_recall_at_100=0.074

Artifact

Artifact는 실험의 증거 파일이다. 성능 숫자만 보면 “좋아졌다”까지는 알 수 있지만, 왜 좋아졌는지는 알기 어렵다. 추천시스템에서는 artifact가 특히 중요하다.

  • metrics.csv: 모델별 NDCG, Recall, Precision, coverage
  • summary.json: 데이터 규모, split, feature, 경로 정보
  • week6_user_diagnostics_*.parquet: 유저별 hit, NDCG, source mix
  • week6_segment_summary_*.csv: segment별 평균 성능
  • lgbm_ranker_*.txt, *.pt: 저장된 모델

4. Harness 관점으로 보는 MLflow

MLflow를 “실험 결과판”으로만 보면 반쪽짜리 이해다. 더 중요한 관점은 MLflow가 머신러닝 실험 harness의 관측 장치라는 점이다.

Harness란 무엇인가

소프트웨어 테스트에서 test harness는 테스트 대상 코드를 감싸고, 입력을 넣고, 실행하고, 결과를 검증하는 장치다. 머신러닝에서도 비슷한 구조가 필요하다. 모델 함수 하나만 있다고 실험이 되는 것이 아니다. 데이터 split, feature 생성, 후보 생성, 학습, 평가, 결과 저장이 모두 한 흐름으로 묶여야 한다.

ML 실험 harness 데이터와 모델을 같은 규칙으로 실행하고, 결과를 같은 형식으로 남기는 반복 실행 장치다.

MLflow는 harness 자체인가?

정확히 말하면 MLflow는 harness 전체가 아니다. Harness는 우리가 작성한 실험 스크립트와 파이프라인이다. MLflow는 그 harness 안에서 실행 기록을 수집하고 보여주는 역할을 한다.

역할 이 저장소의 예시 MLflow와의 관계
Harness 실행기 scripts/week6_two_stage_v2.py 실험을 실제로 실행한다.
데이터 기준 history / rank-label / test split exp_data_* param으로 기록한다.
실험 스위치 candidate cap, related source, ranker params exp_candidate_*, exp_ranker_* param으로 기록한다.
검증 기준 NDCG@K, Recall@K, unique@K metric으로 기록한다.
진단 자료 user diagnostics, segment summary artifact와 segment metric으로 기록한다.

Harness 관점이 중요한 이유

모델 실험은 실험 조건이 조금만 달라도 비교가 불공정해진다. 예를 들어 candidate set이 달라진 상태에서 LGBM과 DeepFM을 비교하면, ranker 성능 차이인지 후보 품질 차이인지 알 수 없다. Harness 관점은 이런 혼동을 줄인다.

  • 같은 split을 쓰는가?
  • 같은 candidate set에서 ranker만 바꿨는가?
  • 같은 metric으로 평가했는가?
  • 실패한 run도 남겼는가?
  • 평균 성능뿐 아니라 segment별 손익도 봤는가?

MLflow는 이 질문에 답하기 위한 화면을 제공한다. 그래서 실무에서는 MLflow를 단순한 로깅 라이브러리가 아니라, 실험 harness를 운영 가능하게 만드는 인터페이스로 보는 편이 낫다.

5. 추천시스템 실험에서 MLflow가 특히 중요한 이유

추천시스템은 일반적인 분류 모델보다 실험 조건이 복잡하다. 모델 하나만 바꾸는 일이 아니라, 후보 생성 방식과 랭킹 방식, 평가 기준이 모두 얽혀 있다.

추천 실험은 보통 두 단계다

1. Data 유저 행동 로그를 시간 기준으로 나눈다.
2. Retrieval ALS, related, recent, popular로 후보를 만든다.
3. Ranking LGBM, DeepFM, DLRM으로 후보 순서를 바꾼다.
4. Evaluation NDCG, Recall, coverage를 계산한다.
5. Diagnosis 유저군과 source별로 원인을 본다.

추천 실험에서 반드시 남겨야 하는 것

범주 기록할 값 MLflow key 예시
데이터 기간 history, rank-label, test 기간 exp_data_history_start, exp_data_test_end
후보 생성 candidate K, hybrid extra, related cap exp_candidate_k, exp_candidate_related_cap
랭커 설정 모델 종류, LGBM leaf, epoch, batch size exp_ranker_names, exp_ranker_lgbm_num_leaves
주요 성능 상단 품질과 후보 회수 primary_ndcg_at_10, primary_recall_at_100
진단 성능 push, PR, high-activity 유저군의 성능 segment_dominant_event_push_ts_ndcg_at_10
주의할 점 추천에서는 평균 NDCG 하나만 보면 위험하다. 전체 평균이 좋아져도 특정 유저군에서 나빠질 수 있고, 인기 repo 몇 개에만 추천이 몰리면 coverage가 나빠질 수 있다.

6. 이 저장소에서 MLflow 보기

이 저장소는 MLflow tracking URI로 sqlite:///mlflow.db를 사용한다. 즉 run 기록은 repo root의 mlflow.db에 저장된다.

MLflow UI 실행

cd bda-2
uv run mlflow ui --backend-store-uri sqlite:///mlflow.db --port 5000

실행 후 브라우저에서 http://127.0.0.1:5000을 열면 된다.

주요 experiment

Experiment 용도 주로 볼 것
bda-week6-two-stage ALS retrieval + LGBM ranking 실험 candidate source, related cap, segment diagnostics
bda-week7-recsys-rankers LGBM/FM/Deep&Wide/DeepFM/DLRM ranker 비교 같은 후보셋에서 ranker별 NDCG/Recall 비교

실험 스크립트

# two-stage 추천 실험
uv run python scripts/week6_two_stage_v2.py \
  --smoke \
  --save-user-diagnostics \
  --output-suffix mlflow_onboarding_smoke

# ranker 비교 실험
uv run python scripts/week6_neural_rankers.py \
  --smoke \
  --rankers lgbm \
  --output-suffix mlflow_onboarding_ranker_smoke

두 스크립트 모두 기본적으로 MLflow에 기록한다. 기록을 끄고 싶을 때만 --no-mlflow를 붙인다.

현재 repo의 MLflow 설계

전체 args는 그대로 남기되, 수업과 실험 비교에서 자주 보는 핵심 설정은 exp_* prefix로 한 번 더 기록한다. 이렇게 하면 MLflow UI에서 열을 고르거나 필터를 걸 때 훨씬 읽기 쉽다.

추천 실험에서 대표로 볼 key는 다음과 같다.

exp_data_* exp_candidate_* exp_ranker_* primary_ndcg_at_* primary_recall_at_* segment_*

7. MLflow UI 읽는 법

MLflow UI를 처음 열면 run 목록이 표로 보인다. 이 표를 “실험 비교표”로 읽으면 된다.

Step 1. Experiment 선택

왼쪽에서 bda-week6-two-stage 또는 bda-week7-recsys-rankers를 선택한다. 지금 보고 싶은 질문이 후보 생성인지, ranker 비교인지에 따라 experiment를 고른다.

Step 2. 필요한 column 켜기

run table에서 column을 조정해 다음 값을 보이게 한다.

  • params.exp_candidate_k
  • params.exp_candidate_related_cap
  • params.exp_ranker_names
  • metrics.primary_ndcg_at_10
  • metrics.primary_recall_at_100
  • metrics.elapsed_min

Step 3. 비교 질문을 명확히 하기

질문 고정할 것 바꿀 것 볼 metric
related source가 도움이 됐나? split, ALS/LGBM 설정 exp_candidate_related_cap primary_ndcg_at_10, primary_recall_at_100, coverage
LGBM 튜닝이 도움이 됐나? candidate set LGBM params primary_ndcg_at_10, primary_ndcg_at_100
neural ranker가 LGBM보다 좋은가? candidate cache, split exp_ranker_names 모델별 ndcg_at_100, recall_at_100
어떤 유저군이 약한가? best run segment segment_* metric, diagnostics artifact

Step 4. Run detail 보기

특정 run을 클릭하면 params, metrics, artifacts가 따로 보인다. 여기서 artifact를 열어 summary JSON, metrics CSV, segment summary CSV를 확인한다.

8. 추천 실험 workflow

좋은 실험은 아무 모델이나 여러 번 돌리는 것이 아니다. 비교 질문을 정하고, 그 질문에 필요한 부분만 바꾸는 방식으로 진행한다.

권장 흐름

  1. 기준 run을 하나 정한다.
  2. 이번 실험의 질문을 한 문장으로 쓴다.
  3. 고정할 조건과 바꿀 조건을 분리한다.
  4. 스크립트를 실행해 MLflow run을 남긴다.
  5. primary metric으로 1차 비교한다.
  6. artifact와 segment metric으로 원인을 확인한다.
  7. 개선이 맞으면 다음 기준 run으로 승격한다.

예시 1. Candidate source 실험

uv run python scripts/week6_two_stage_v2.py \
  --best-full \
  --related-candidate-cap 80 \
  --related-top-per-anchor 10 \
  --related-max-seen-anchors 20 \
  --save-user-diagnostics \
  --output-suffix related80_anchor20_full

이 실험의 질문은 “같은 유저가 함께 본 repo를 related source로 추가하면 후보 회수와 상단 품질이 좋아지는가?”다. 따라서 MLflow에서는 related cap과 Recall@100, NDCG@10, segment metric을 함께 본다.

예시 2. Ranker 비교 실험

uv run python scripts/week6_neural_rankers.py \
  --reuse-feature-cache \
  --feature-cache-path data/models/week6/week6_ranker_compare_week7_full_500k_u100k_e3_features.pkl \
  --rankers lgbm,fm,deepwide,deepfm,dlrm \
  --epochs 3 \
  --batch-size 32768 \
  --torch-threads 1 \
  --device cpu \
  --output-suffix week7_ranker_compare

이 실험의 질문은 “같은 후보셋에서 어떤 ranker가 가장 좋은가?”다. 후보 cache를 재사용해야 ranker 비교가 공정해진다.

좋은 run 이름

run 이름은 나중에 봐도 실험 의도가 보여야 한다.

  • 좋음: related80_anchor20_full_als96_i12_lgbm63
  • 좋음: week7_full_500k_u100k_e3_reuse_b32768
  • 나쁨: test, final, new_new_final

9. 실험 전후 체크리스트

실험 전

  • 이번 실험 질문이 한 문장으로 말해지는가?
  • 비교할 baseline run이 정해져 있는가?
  • 데이터 split이 이전 run과 같은가?
  • 후보 생성 실험인지, ranker 실험인지 구분했는가?
  • run 이름이 나중에 봐도 이해되는가?

실험 후

  • MLflow run status가 FINISHED인지 확인했는가?
  • primary_ndcg_at_10과 primary_recall_at_100을 함께 봤는가?
  • coverage 또는 unique recommended가 무너졌는가?
  • segment metric에서 특정 유저군이 나빠졌는가?
  • summary JSON과 metrics CSV artifact가 남았는가?
  • 좋아진 이유를 한 문장으로 설명할 수 있는가?
취준생 포인트 포트폴리오에서는 “NDCG가 올랐다”보다 “후보 source를 추가했고, Recall@100과 coverage가 같이 좋아졌으며, push/PR segment에서도 손실이 줄었다”가 훨씬 강한 설명이다.

10. FAQ

Q. MLflow가 모델 성능을 올려주나요?

직접 올려주지는 않는다. 대신 어떤 변경이 성능을 올렸는지, 어떤 변경이 실패했는지 빨리 알게 해준다. 이게 반복되면 결과적으로 모델 개선 속도가 빨라진다.

Q. 그냥 CSV로 기록하면 안 되나요?

작은 실험은 CSV로도 가능하다. 하지만 run이 늘어나면 params, metrics, artifacts, git 정보, 모델 파일이 흩어진다. MLflow는 이들을 run 단위로 묶어준다.

Q. 모든 값을 metric으로 올리면 좋나요?

아니다. UI에서 비교할 핵심 숫자는 metric으로 올리고, 큰 표나 유저별 결과는 artifact로 남기는 편이 좋다. 예를 들어 유저별 diagnostics 전체는 parquet artifact로 두고, 대표 segment 평균만 metric으로 올린다.

Q. 추천시스템에서 가장 먼저 봐야 할 metric은 무엇인가요?

상단 품질은 NDCG@10, 후보 회수는 Recall@100, 쏠림 여부는 unique@100을 함께 본다. 하나만 보면 착시가 생긴다.

Q. Harness 관점에서 MLflow를 한 문장으로 말하면?

MLflow는 머신러닝 실험 harness가 남기는 실행 기록을 표준화해서, 같은 질문을 같은 기준으로 반복 검증할 수 있게 해주는 도구다.
