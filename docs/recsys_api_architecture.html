Bundle은 왜 필요한가

추천 API는 model.pkl 하나만으로 동작하지 않는다. 먼저 추천 후보를 찾고, 후보별 feature를 만들고, 그 feature를 ranker에 넣어 최종 순위를 계산한다.

bundle은 이 파일들이 서로 맞는 버전인지 기록한 체크리스트다. 즉 “이 candidate 파일과 이 feature 규칙으로 검증된 모델만 API에 올린다”는 약속이다.

모델 파일만 바꾸면 생길 수 있는 일 새 ranker는 candidate_source_code feature를 기대하는데 API가 예전 candidate 파일을 읽으면, 추천 점수가 깨지거나 에러가 난다. bundle은 이런 mismatch를 줄이기 위한 최소 장치다.
파일 버전 맞추기 실험 결과 추적 배포 실수 줄이기 이전 버전 복구

recsys_v2_20260502_hybrid_lgbm_n20 안에 묶인 것

Canonical data 유저가 과거에 본 repo와 평가용 split의 기준 데이터
Candidate cache ranker가 점수를 매길 actor별 후보 repo 목록
LGBM ranker 후보 repo를 1등, 2등, 3등 순서로 다시 정렬하는 모델
Feature contract 모델 입력 컬럼 이름과 순서가 맞는지 확인하는 규칙
Eval metrics 이 조합이 얼마나 좋았는지 비교하는 rerank.ndcg@100 같은 점수
Promotion state 여러 실험 중 지금 API가 써야 하는 버전 표시

추천 요청 처리 흐름

1. 요청 POST /v1/recsys/recommendations
actor_id, k, bundle_id optional
2. Bundle 결정 bundle_id=null이면 recsys_promoted.json의 active bundle 사용
3. Candidate 조회 parquet filter로 actor_id 후보만 읽고 candidate_rank 기준 정렬
4. Feature + Rerank history 통계 feature 생성 후 LGBM score 내림차순 정렬
5. 응답 repo_id, rank, score, candidate_source, metadata 반환

API 계약

  • GET /health 서비스 상태와 promoted bundle id 확인
  • GET /v1/recsys/bundles/active 현재 API가 서빙하는 manifest 조회
  • POST /v1/recsys/bundles/{id}/promote artifact 존재 검증 후 active bundle 교체
  • GET /v1/recsys/users/{actor_id}/candidates rerank 전 candidate/source 분포 디버깅
Registry boundary API는 latest 파일을 직접 고르지 않는다. bundle manifest와 explicit promotion이 운영 기준이다.
Runtime boundary Docker는 torch, CUDA, implicit 없이 serving에 필요한 FastAPI, pandas, pyarrow, LightGBM 중심으로 구성했다.
Verification boundary ruff, pytest, health, actor smoke test, docker build/up까지 확인된 상태를 기준으로 설명한다.
