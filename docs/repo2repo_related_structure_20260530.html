연관 추천 구조: repo2repo 후보가 개인화 랭킹에 들어가는 방식

5/30 기준 실험은 related 후보 생성기를 바꿔가며 같은 ALS base, 같은 user sample, 같은 LGBM LambdaRank 조건에서 비교했다.

기준일: 2026-05-30 최종 지표: NDCG@100 test user sample: 100k actual eval users: 57,616 candidate_k: 300 ranker: LGBM LambdaRank

한 줄 결론

이번 승자

rule_related NDCG@1000.007622

단순 co-watch rule이 LGBM 개인화 랭킹까지 통과했을 때 가장 좋았다.

ML related의 현재 상태

cooc_norm0.004981
als_item_cosine0.004888
item2vec_svd0.003274

해석

repo2repo 자체 평가는 cooc가 강하지만, 최종 개인화 랭커에서는 source mix, candidate rank, seen/history 피쳐와의 상호작용 때문에 rule이 더 잘 먹혔다.

성과 정의: 두 문제를 분리해서 봐야 함

문제 A. 연관 추천 자체

질문: 사용자가 repo A를 봤다면, 그 다음 기간에 같이 볼 repo B를 잘 맞히는가?

입력: anchor_repo = history에서 본 repo A
정답: label 기간에 같은 사용자가 본 future target repo B
평가: repo A별 top-k related list가 B를 얼마나 위에 올리는지
주요 지표NDCG@10/50, Recall@100
여기서 co-occurrence는?성과가 아니라 후보 생성 신호

문제 B. 개인화 랭킹 성과

질문: related 후보를 user별 추천 후보 풀에 넣었을 때, 최종 추천 100개가 test 정답을 더 잘 맞히는가?

입력: user history + ALS/recent/popular/related candidates
정답: test 기간에 사용자가 본 repo
평가: LGBM rerank 이후 user별 top-100 추천 품질
주요 지표NDCG@100
이번 결론ML related lift 증명 안 됨
정리하면 co-occurrence는 "연관 추천을 만드는 방법" 중 하나다. 성과는 co-occurrence 값 자체가 아니라, 그 값으로 정렬한 related list가 미래 행동을 맞히는지로 정의한다.

이번 실험에서 새로 봐야 할 용어

용어 쉬운 설명 이번 실험에서의 뜻
추천 사용자가 다음에 볼 만한 repo를 골라서 위에 올리는 일 user별 top-100 repo 목록 만들기
연관 추천 "이 repo를 본 사람은 저 repo도 자주 봤다"를 이용하는 추천 repo A를 anchor로 두고 관련 repo B를 찾는 것
co-occurrence 두 repo가 같은 사용자에게 같이 등장한 횟수 A와 B를 함께 본 사용자가 많으면 A -> B score가 커짐

예시로 보는 연관 추천

1. 과거 행동을 모은다

사용자 1react, next.js, vite를 봄
사용자 2react, next.js를 봄
사용자 3react, django를 봄

2. repo끼리 연결한다

react -> next.js같이 등장 2번, 강한 관계
react -> vite같이 등장 1번, 약한 관계
react -> django같이 등장 1번, 약한 관계
여기서 중요한 점은 "같이 많이 나왔다"가 끝이 아니라는 것. 진짜 평가는 과거에 react를 본 사용자가 미래 기간에 next.js를 봤는지로 한다.

추천 시스템은 두 단계로 움직임

1단계: 후보 많이 모으기

사용자에게 보여줄 가능성이 있는 repo를 넓게 모은다. 이때 ALS, recent, popular, related가 각각 후보를 낸다.

목표: 정답일 가능성이 있는 repo를 최대한 후보 300개 안에 넣기

2단계: 순서 다시 매기기

300개 후보를 그대로 보여주지 않고, LGBM ranker가 피쳐를 보고 "무엇을 더 위에 둘지" 다시 정한다.

목표: 후보 300개 중 진짜 볼 만한 repo를 top-100 위쪽에 올리기
그래서 related 알고리즘이 좋아 보여도, 최종 개인화 랭커가 그 후보를 위로 안 올리면 개인화 성능은 안 좋아질 수 있다.

숫자는 이렇게 읽으면 됨

NDCG

정답 repo가 추천 목록 위쪽에 있을수록 높은 점수. 같은 정답을 맞혀도 90등보다 3등에 올리면 더 좋게 본다.

Recall

사용자가 실제로 본 정답 repo 중 몇 개를 추천 목록 안에 넣었는지. 순서보다 "넣었냐"에 가깝다.

Coverage / unique

추천이 얼마나 다양한 repo로 퍼지는지. 높으면 다양하지만, 반드시 정확도가 높다는 뜻은 아니다.

개선 해석도 둘로 쪼갬

영역 이번에 개선됐다고 볼 수 있는 것 아직 증명 안 된 것 현재 판단
연관 추천 자체 cooc_norm은 anchor_repo -> future target repo 평가에서 강함. ALS/item2vec은 coverage를 넓힘. embedding/hybrid가 cooc_norm보다 top-rank 품질을 이긴다는 증거 repo2repo graph 품질은 cooc_norm이 현재 제일 방어 가능
개인화 랭킹 related 후보를 넣는 것 자체는 related_off 대비 lift가 큼. 특히 rule_related가 최고. ML related, ALS cosine, item2vec, hybrid가 rule_related보다 좋다는 증거 개인화에서는 ML 도입 성과 없음. 룰 기반 related/source 피쳐 효과로 보는 게 맞음.

End-to-End Pipeline

1. Canonical split watch/star/fork 등 positive interaction을 시간순으로 나누고 repo/user filter를 적용한다.
2. ALS base retrieval user history 기반 ALS 후보를 먼저 만들고 head/tail slot으로 300개 후보 골격을 채운다.
3. Related graph anchor repo별 future target repo 후보를 rule, cooc, ALS cosine, hybrid, item2vec 방식으로 만든다.
4. Candidate merge user가 본 repo들의 related 후보를 펼쳐 candidate_source, source_rank, source_score를 붙인다.
5. LGBM rerank rank_label로 LambdaRank를 재학습하고 test에서 NDCG@100으로 최종 비교한다.

알고리즘

방법 무엇을 학습/계산하나 score 의미 강점과 리스크
related_off related 후보를 끄고 ALS/recent/popular만 사용 없음 관련 추천의 incremental lift를 보는 baseline
rule_related 같은 사용자가 history 기간에 함께 본 repo pair를 누적 co-watch 강도와 repo popularity를 섞은 rule score 단순하지만 현재 최종 랭킹에서 가장 강함. 인기 repo 쏠림은 관리 필요
cooc_norm anchor-target 동시 등장 횟수를 item frequency로 정규화 인기 보정된 related strength repo2repo 자체 NDCG는 최고. 랭커 feature와 결합 시 rule보다 낮음
als_item_cosine user-repo implicit matrix로 ALS item factor 학습 후 item cosine top-k 검색 embedding 공간의 repo 유사도 coverage가 넓고 tail 탐색에 좋음. top-rank 정밀도는 rule/cooc보다 낮음
hybrid_rule_als cooc 계열 score와 ALS item similarity를 정규화 후 가중합 w * cooc + (1-w) * als coverage와 precision을 절충. 현재 sweep으로 최적 weight 재검증 중
item2vec_svd repo co-occurrence matrix를 PPMI 성격으로 만들고 SVD embedding 근사 분포적 item embedding 유사도 빠른 실험용 baseline. full item2vec이 아니어서 ALS와 차별점이 약함

rule/cooc 계열 직관

사용자 history 안에서 repo A와 repo B가 같이 나타나면 A -> B edge를 만든다. 많이 같이 나오면 강하지만, 너무 인기 있는 repo는 정규화로 눌러준다.

cooc_norm(A,B) ~= cooc(A,B) / sqrt(users(A) * users(B))

embedding 계열 직관

ALS/item2vec은 repo를 벡터로 바꾸고 가까운 repo를 찾는다. 직접 동시 등장하지 않은 repo도 연결할 수 있어 coverage는 좋지만, top slot 정밀도는 별도 튜닝이 필요하다.

related(A) = topK cosine(item_vector[A], item_vector[*])

co-occurrence는 어떻게 계산하나

1. 사용자별 repo 묶음을 만든다

history 기간에 한 사용자가 본 repo들을 하나의 묶음으로 본다. 같은 묶음 안에 같이 등장한 repo pair를 모두 만든다.

user1A, B, C를 봄 -> A-B, A-C, B-C pair 생성
user2A, B를 봄 -> A-B pair 생성
결과A-B는 2번, A-C는 1번 같이 등장

2. 너무 인기 많은 repo를 보정한다

그냥 같이 등장한 횟수만 쓰면 React, Linux처럼 원래 많이 보는 repo가 항상 이긴다. 그래서 각 repo의 전체 등장량으로 나눠서 "진짜 관련성"에 가깝게 만든다.

raw cooc(A,B) = A와 B가 같은 사용자 history에 같이 나온 횟수
cooc_norm(A,B) ~= raw cooc(A,B) / sqrt(users(A) * users(B))

cooc의 장점

설명하기 쉽고, 실제 행동 데이터에 바로 붙어 있다. "A를 본 사람이 B도 봤다"라는 근거가 명확해서 repo2repo 자체 평가에서는 강하게 나온다.

cooc의 약점

같이 등장한 기록이 적은 새 repo나 tail repo에는 약하다. 또 "같이 봤다"는 사실만으로 왜 관련 있는지는 알 수 없다.

item2vec은 무엇이 다른가

직관

item2vec은 word2vec의 아이디어를 repo에 적용한 것이다. 문장에서 주변 단어가 비슷하면 단어 의미가 비슷하다고 보는 것처럼, 사용자 history 안에서 주변 repo가 비슷하면 repo 성격이 비슷하다고 본다.

word2vec: "나는 커피를 마셨다" 안에서 단어 관계 학습
item2vec: "user가 A, B, C repo를 봤다" 안에서 repo 관계 학습

이번 구현

이번에는 full word2vec 학습을 돌린 게 아니라, item co-occurrence 행렬을 만들고 SVD로 embedding을 근사했다. 그래서 엄밀히 말하면 "item2vec"이라기보다 co-occurrence matrix factorization baseline에 가깝다.

user history -> repo co-occurrence matrix -> PPMI 성격의 보정 -> SVD -> repo embedding -> cosine top-k

왜 쓸 만한가

cooc는 직접 같이 등장한 pair에 강한 반면, item2vec은 비슷한 문맥을 통해 간접 관계도 잡을 수 있다. 예를 들어 A와 D가 직접 같이 많이 안 나와도, 둘 다 B/C와 자주 같이 나오면 가까운 repo로 볼 수 있다.

AB, C와 자주 같이 등장
DB, C와 자주 같이 등장
해석A와 D가 직접 같이 적게 나와도 비슷한 위치의 repo일 수 있음

이번 결과가 낮은 이유

이번 item2vec_svd는 빠른 검증용이라 user cap이 작고, full SGNS 방식이 아니었다. 더 중요한 점은 SVD로 co-occurrence 행렬을 줄이는 방식이면 ALS와 문제 설정이 많이 겹친다는 것이다. 이 조건에서는 굳이 item2vec_svd를 별도 주력 알고리즘으로 볼 이유가 약하고, ALS가 더 안정적인 선택이다.

이번 결과: item2vec_svd NDCG@100 = 0.003274
rule_related NDCG@100 = 0.007622
정리하면 이번 item2vec_svd는 "새로운 강한 알고리즘"이라기보다, item embedding 계열을 빠르게 찍어본 실험이다. 현재 결과 기준으로는 ALS가 더 낫고, item2vec을 계속 보려면 negative sampling 기반 SGNS로 다시 구현해야 의미가 있다.

좋아진 부분

related_off는 연관 추천을 끈 상태다. 여기서 rule_related를 켜면 NDCG@100이 0.002638에서 0.007622로 오른다. 즉 "연관 후보를 넣는 것" 자체는 효과가 있다.

related 후보 없음 -> 0.002638
rule related 후보 있음 -> 0.007622

좋아졌다고 말하면 안 되는 부분

ML 방식인 cooc_norm, ALS cosine, hybrid, item2vec이 rule_related를 이기지 못했다. 그래서 "ML related를 도입해서 개인화가 좋아졌다"는 결론은 아직 아니다.

왜 rule이 ML을 이겼나

1. 최종 목적은 graph 품질이 아님

repo2repo NDCG가 높아도, 사용자별 후보 풀 안에서 LGBM이 score를 어떻게 해석하느냐가 최종 성능을 결정한다.

2. slot/source 효과

candidate_source_code, candidate_rank, source_rank가 강한 피쳐라서 같은 related라도 source distribution이 달라지면 랭킹 결과가 크게 바뀐다.

3. embedding은 넓고 얕음

ALS cosine은 unique@100이 가장 높다. 하지만 NDCG는 상위권 정답을 빨리 올려야 하므로 coverage만으로는 부족하다.

발표용 결론 문장

쉬운 버전: 이번 실험에서는 "이 repo를 본 사람은 저 repo도 봤다"는 단순한 연관 규칙을 추천 후보에 추가했을 때 개인화 추천 성능이 가장 좋아졌다. 반면 ALS, hybrid, item2vec 같은 ML 기반 연관 추천은 아직 룰 기반 방식을 넘지 못했다.

따라서 이번 성과는 "ML 연관 추천 도입 성공"이라기보다 "룰 기반 연관 후보와 관련 피쳐를 개인화 랭커에 넣은 효과"로 보는 것이 맞다.
