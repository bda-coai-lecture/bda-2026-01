핵심 결론

Retrieval

ALS는 interaction-only 기준선인데도 NDCG@10에서 Two-Tower보다 높았다. Two-Tower는 대신 추천 다양성이 매우 커서 후보 다양화 장치로 보는 게 맞다.

Re-rank

같은 후보군 위에서는 LGBM이 1위였다. 최신 run에서는 neural ranker들이 같은 순위로 수렴해서, 모델 우열보다 학습/튜닝 한계로 보는 게 맞다.

데이터 해석

positive rate가 0.537%라서 deep model이 충분히 학습하기 어렵다. 반면 popularity, recent growth, ALS score 같은 수작업 feature는 즉시 강한 신호가 된다.

이번 구현에서 실제로 반영된 부분 Two-Tower는 repo metadata와 popularity proxy를 tower 입력에 넣고 in-batch negative로 retrieval embedding을 학습했다. re-ranker 쪽은 모두 같은 candidate cache와 같은 75개 feature를 공유하게 만들고, 모델 구조만 LGBM, FM, Deep&Wide, DeepFM, DLRM으로 바꿔 비교했다. 그래서 이 문서에서는 “논문 모델을 완전 재현했다”가 아니라 “우리 추천 파이프라인의 어느 단계에 어떤 아이디어를 끼워 넣었는가”로 설명하는 게 정확하다.

NCF

NCF는 user embedding과 item embedding의 단순 내적 대신 MLP가 interaction function을 학습한다. 현재 문서에서는 “왜 딥러닝 추천이 등장했는가”를 설명하는 배경으로만 다루고, 실험 성능 비교에는 포함하지 않는다.

Two-Tower: 후보를 넓히는 retrieval 모델

Two-Tower는 user tower와 item tower를 따로 학습해 같은 embedding 공간에 올린 뒤, inner product나 cosine similarity로 빠르게 nearest neighbor search를 하는 구조다. DSSM 계열의 query-document matching 아이디어를 추천의 user-repo matching으로 옮긴 형태로 보면 된다.

적용은 scripts/train_two_tower_week6_full_v2.py에서 했다. user feature는 total score, active days, event share, entropy, growth를 쓰고, item feature는 7/28/42일 repo 활동량, language, stars, forks, archived 등을 썼다. 학습은 batch 내부의 positive item을 정답으로 두는 in-batch softmax 방식이며, temperature는 0.05, embedding dim은 64다.

Positive item 정의

Two-Tower에서 positive item은 해당 user가 history 기간에 실제 GitHub 이벤트를 남긴 repo다. train_fb의 각 (actor_id, repo_id) row가 positive pair가 되고, 이 pair는 watch, fork, PR, issue, push, comment 같은 행동을 weighted implicit feedback으로 바꾼 결과다.

observed interactionimplicit positive

Negative sample 방식

별도 negative sampling table을 만들지는 않았다. batch 안에서 i번째 user의 정답은 i번째 item이고, 나머지 item들은 그 user 입장에서 negative처럼 쓰인다. 즉 batch_size=2048이면 user당 최대 2047개의 in-batch negative가 생긴다.

in-batch negativefalse negative 가능
Popularity는 어떻게 반영됐나 인기 repo는 여러 user의 positive pair로 더 자주 등장하기 때문에 item embedding이 더 많이 업데이트된다. 동시에 v2 item tower에는 total_score_7d/28d/42d, unique_users_7d/28d/42d, stars, forks 같은 popularity proxy가 feature로 직접 들어간다. 그래서 popularity는 학습 row 빈도와 item feature 두 경로로 반영됐다.
Two-Tower가 특히 강해지는 구간 Two-Tower는 초기 유저나 정보가 적은 유저, 그리고 새로 들어온 item처럼 cold-start가 중요한 상황에서 강해질 수 있다. ALS는 충분한 user-item interaction이 쌓여야 유저/아이템 latent vector가 안정적으로 잡히지만, Two-Tower는 연령대, 성별, 지역, 선호 카테고리 같은 user metadata와 카테고리, 브랜드, 가격대, 텍스트/이미지 embedding 같은 item metadata를 tower 입력에 바로 넣을 수 있다. 그래서 interaction이 적어도 metadata가 풍부하면 “비슷한 프로필의 유저가 좋아한 item”이나 “비슷한 속성의 item”을 retrieval 단계에서 꺼내기 쉽다. 이번 GitHub 데이터에서는 개인정보나 상품 카테고리처럼 강한 side feature가 없고, repo metadata도 language, stars/forks, 활동량 중심이라 이 장점이 제한적으로만 드러났다.

이커머스 예시

새 유저가 회원가입 직후 “Python, 데이터, 저가형 노트북” 같은 관심사만 남겼다고 하자. ALS는 아직 구매/클릭 history가 부족해서 유저 벡터가 약하지만, Two-Tower는 이 관심사를 user feature로 넣고 item tower의 카테고리, 가격대, 브랜드, 텍스트 embedding과 바로 매칭할 수 있다.

우리 데이터에서의 대응

GitHub에서는 user profile이 거의 없으므로 user tower는 이벤트 비중, 활동일수, growth 같은 행동 요약에 의존했다. item tower는 language, stars, forks, 최근 활동량을 썼다. 즉 cold-start 장점은 넣었지만, 이커머스만큼 풍부한 metadata는 없었다.

참고로 notebooks/ghrec/09_two_tower.ipynb의 5% sample 실험에서는 Two-Tower가 ALS보다 높게 나온 구간도 있었다. 하지만 2026-05-16 기준 최신 feature와 100k item catalog로 다시 비교했을 때는 ALS가 NDCG 기준으로 앞섰고, Two-Tower는 다양성 보강 역할로 결론이 바뀌었다.

왜 ALS보다 낮았나 Two-Tower는 side feature를 넣을 수 있지만, 이 데이터에서는 repo interaction이 극도로 sparse하고 인기 repo로 신호가 쏠린다. ALS는 바로 user-item interaction 행렬을 factorize하기 때문에 warm user의 협업 신호를 더 직접적으로 잡는다. Two-Tower의 장점은 정확도 1위가 아니라 Unique@100 70,537개로 후보 다양성을 크게 늘리는 쪽이었다.

Neural re-ranker: 같은 후보군을 더 잘 정렬하기

re-rank 실험은 ALS/popularity/recent/related 기반 hybrid candidate를 만든 뒤, 75개 feature를 사용해 후보 순서를 다시 매겼다. 딥러닝 모델은 후보 생성기가 아니라 후보 정렬기로 넣었다. 즉 모델이 모든 repo를 직접 검색하는 것이 아니라, 이미 좁혀진 후보 300~500개 안에서 “이번 user에게 어떤 repo를 더 위에 둘 것인가”를 학습한다.

공통 입력 모든 re-ranker는 같은 75개 feature를 받는다. 주요 feature는 als_score, factor_cosine, log_popularity, log_recent_popularity, pop_growth, repo event share, user event share, event_match_dot, seen_max_cosine, 그리고 후보 출처 flag(source_is_als, source_is_recent, source_is_popular)다.
학습 loss와 평가 지표는 다르다 LGBM은 metric=ndcg로 평가하지만 실제 objective는 lambdarank다. 후보 pair의 순서를 바꾸는 방향으로 학습하되, 그 pair가 NDCG에 미치는 영향이 클수록 더 큰 gradient를 준다. 반면 FM, Deep&Wide, DeepFM, DLRM은 모두 BCEWithLogitsLoss(pos_weight=185.32)로 학습했다. 즉 후보별 positive 확률을 맞추는 binary classification loss로 학습한 뒤, inference 때 logit score로 후보를 정렬해 NDCG@K와 Recall@K를 평가했다. 이 차이 때문에 LGBM은 Top-K ranking 목적과 더 직접적으로 맞고, neural ranker들은 NDCG와 objective 사이에 간극이 있다.

LGBM LambdaRank: tabular ranking 기준선

LGBM은 딥러닝 모델은 아니지만 re-rank 비교의 가장 중요한 기준선이다. LambdaRank objective는 같은 user의 후보 리스트를 하나의 group으로 보고, 정답 repo가 위로 올라갈 때 NDCG가 얼마나 좋아지는지를 학습 신호로 반영한다. 그래서 후보별 독립 분류보다 Top-K ranking metric과 더 직접적으로 맞는다.

0.022160 NDCG@100, 전체 1위
score = sum(tree_t(feature_vector, split_rules))
핵심 아이디어

feature threshold 조합을 tree split으로 찾는다. 예를 들어 ALS score가 높고 recent popularity도 높은 후보를 별도 leaf로 보낼 수 있다.

우리 적용

objective=lambdarank, n_estimators=120, learning_rate=0.05, num_leaves=31.

왜 강했나

positive label이 0.24%뿐이라 deep model이 interaction을 충분히 배우기 어렵다. LGBM은 적은 label에서도 강한 tabular rule을 빨리 잡는다.

DLRM-style: field embedding과 pairwise dot interaction

DLRM은 Meta/Facebook의 추천 모델로, sparse categorical field는 embedding table로 바꾸고 dense feature는 MLP로 처리한 뒤 field embedding 사이의 pairwise dot product를 만든다. 추천/광고처럼 field가 많고 interaction이 중요한 문제에 맞춘 구조다. 우리 구현은 원본 DLRM을 그대로 재현한 것이 아니라, 75개 dense scalar feature를 각각 하나의 field로 보고 embedding_dim=16 공간에 투영한 DLRM-style ranker다.

0.015860 NDCG@100, neural ranker 공통값
field_i = x_i * W_i + b_i, interaction_ij = dot(field_i, field_j), logit = MLP([x, interactions])
핵심 아이디어

feature pair를 그냥 곱하지 않고, 각 feature를 embedding field로 올린 뒤 dot product로 관계를 표현한다.

우리 적용

75개 feature면 pairwise interaction은 75 * 74 / 2 = 2775개다. 여기에 원 feature 75개를 붙여 MLP에 넣는다.

성능 해석

최신 run에서는 다른 neural ranker와 같은 순위로 수렴했다. 구조 자체는 feature pair를 명시적으로 보지만, 이번 학습 budget에서는 우열을 만들 만큼 분화하지 않았다.

쉽게 보면: “조합 점수표”를 자동으로 만드는 모델

단일 feature만 보면 recent_popularity가 높은 repo는 대체로 좋아 보인다. 하지만 어떤 유저에게는 “최근 뜬 repo”보다 “내가 자주 쓰는 language와 맞고, 이미 본 repo와 embedding이 가까운 repo”가 더 중요할 수 있다. DLRM-style interaction은 이런 최근성 x 언어/행동 일치 x 후보 출처 조합을 496개 pair score로 펼쳐서 MLP가 보게 만든다.

코드에 반영된 부분

DenseDLRM은 원 feature 75개를 그대로 버리지 않고, 각 scalar를 field_weight와 field_bias로 16차원 field embedding에 투영한다. 그다음 upper-triangle dot product만 뽑아 원 feature와 concat한다. 이 구현은 원 논문식 sparse embedding lookup 대신 dense feature field projection을 쓴 변형이다.

DLRM이 보통 성공적인 조건 DLRM은 광고 CTR, 이커머스 추천, 앱 추천처럼 user, item, context categorical feature가 많고 cardinality가 큰 환경에서 특히 잘 맞는다. 예를 들어 user id, item id, 카테고리, 브랜드, 지역, 디바이스, 시간대, 캠페인, 세션 정보 같은 field가 많고, 20대 남성 x 게임 카테고리, 서울 x 점심시간 x 배달 카테고리, 최근 fork 많이 한 유저 x fast-growing repo처럼 feature 조합이 클릭/구매/interaction 확률을 좌우할 때 강하다. 보통은 retrieval이 후보 수백~수천 개를 먼저 줄이고, DLRM이 그 후보들을 user-item-context feature로 re-rank하는 구조에서 쓰인다.
우리 실험과 원래 DLRM setting의 차이 원래 DLRM은 많은 sparse categorical field를 embedding table로 처리하는 모델이다. 이번 프로젝트의 구현은 원본 DLRM을 그대로 재현한 것이 아니라, 이미 만들어진 75개 dense candidate feature를 각각 field처럼 보고 projection한 DLRM-style 모델이다. 따라서 거대한 user/item/category embedding table을 학습한 것은 아니고, positive label도 877개로 많지 않았다. 최신 run에서 DLRM이 다른 neural ranker와 같은 결과로 수렴한 것은 event_match_dot, pop_growth, source_is_als, recent_popularity, user activity 같은 feature 조합을 충분히 분리해 학습하기에는 label과 tuning budget이 부족했기 때문으로 해석하는 편이 안전하다.

Deep&Wide: 외운 규칙과 일반화 규칙을 같이 쓰기

Wide & Deep은 “자주 본 조합은 wide linear path로 외우고, 못 본 조합은 deep MLP로 일반화한다”는 구조다. 원래 논문은 app recommendation에서 cross-product feature와 embedding을 함께 쓰지만, 여기서는 이미 만든 75개 dense feature를 wide path와 deep path에 동시에 넣었다.

0.015860 NDCG@100, DeepFM과 사실상 동률
logit = Linear(x) + MLP_128_64(x)
핵심 아이디어

wide path는 강한 단일 feature 신호를 보존하고, deep path는 비선형 조합을 학습한다.

우리 적용

MLP는 Linear -> ReLU -> BatchNorm1d -> Dropout을 두 번 쌓고 hidden dim은 128,64다.

성능 해석

최신 run에서는 FM/DeepFM/DLRM과 같은 값으로 수렴했다. 구조 차이보다 label 희소성과 공통 BCE 학습 설정의 영향이 더 컸다.

Deep&Wide가 잘 맞는 조건 반복적으로 잘 먹히는 feature rule과 새로운 조합을 동시에 다뤄야 할 때 좋다. 예를 들어 source_is_als=1이고 als_score가 높으면 wide path가 안정적으로 높은 점수를 줄 수 있고, push 중심 유저 x 최근 issue가 늘어난 repo x language match처럼 명시적으로 만들지 않은 조합은 deep path가 배울 수 있다. 우리 구현은 cross-product sparse feature를 별도로 만들지 않았기 때문에, 원 논문보다 wide path의 “암기” 역할은 약하고 dense feature의 linear shortcut에 가깝다.

DeepFM: FM interaction과 deep interaction을 결합

DeepFM은 FM의 low-order interaction과 deep network의 high-order interaction을 합친 모델이다. 원 논문에서는 sparse categorical field embedding을 FM part와 deep part가 공유해서 feature engineering 없이 CTR feature interaction을 학습하는 것이 핵심이다. 우리 구현은 categorical embedding table을 새로 만들지는 않고, 표준화된 dense candidate feature를 FM과 MLP에 같이 넣는 간소화 버전이다.

0.015860 NDCG@100, Deep&Wide와 거의 동일
logit = FM(x) + MLP_128_64(x)
핵심 아이디어

FM이 2차 조합을 안정적으로 담당하고, MLP가 3차 이상의 feature 조합을 담당한다.

우리 적용

DenseFM(n_features, 16)와 make_mlp(75, 128,64)의 출력을 더한다.

성능 해석

Deep&Wide와 거의 같았다. 이미 feature engineering이 강해서 FM part가 추가로 주는 이득이 제한적이었다.

DeepFM이 잘 되는 조건

원래 DeepFM은 광고/이커머스 CTR처럼 user id, item id, 카테고리, 브랜드, context 같은 sparse field가 많고 2차 조합과 고차 조합이 둘 다 중요한 문제에서 강하다. feature engineering 없이 embedding을 공유하며 user segment x category, brand x price bucket x time 같은 조합을 같이 배운다는 점이 핵심이다.

우리 DeepFM의 한계와 해석

이번 구현은 sparse categorical embedding을 새로 만들지 않고, 이미 가공된 75개 dense feature를 FM과 MLP에 넣었다. 그래서 DeepFM의 장점인 “raw sparse field에서 interaction을 직접 학습”하는 효과는 약했다. 성능이 Deep&Wide와 거의 같은 이유도, 두 모델이 같은 dense feature와 같은 MLP 설정을 공유했기 때문이다.

FM: pairwise interaction을 가장 작게 넣는 모델

Factorization Machine은 선형 모델에 feature 간 2차 interaction을 더한다. 모든 feature 쌍마다 별도 weight를 두면 parameter가 너무 많아지므로, 각 feature에 작은 latent vector를 주고 두 vector의 내적으로 interaction strength를 계산한다.

0.015860 NDCG@100, neural 계열 시작점
y = w0 + sum(w_i x_i) + 0.5 * sum((xV)^2 - (x^2)(V^2))
핵심 아이디어

als_score x source_is_als, recent_popularity x pop_growth 같은 2차 조합을 자동으로 잡는다.

우리 적용

factor_dim=16. 입력은 one-hot sparse field가 아니라 표준화된 dense feature 75개다.

성능 해석

최신 run에서는 다른 neural ranker와 같은 값으로 수렴했고 ALS/Fallback보다 낮았다. 2차 interaction만으로는 LGBM의 ranking objective를 넘지 못했다.

왜 neural ranker가 LGBM을 못 넘었나 이번 데이터는 rank row가 16.3만 개이고 positive label은 877개뿐이다. deep model은 많은 negative 사이에서 미세한 ranking 차이를 배워야 하고, weighted BCE는 NDCG 자체를 직접 최적화하지 않는다. 반면 LGBM LambdaRank는 user별 group ranking objective로 학습하고, feature threshold가 강한 GitHub popularity/recentness 패턴에 잘 맞았다.

튜닝한 영역과 남은 선택지

이번 실험은 모든 하이퍼파라미터를 대규모 grid search로 훑은 작업은 아니다. 비교 목적에 맞춰 먼저 같은 candidate set과 같은 feature cache를 고정하고, 모델 구조만 바꿔 공정하게 비교하는 쪽을 우선했다. 그다음 로컬 16GB Mac에서 끝까지 재현 가능한 설정으로 batch size, epoch, cache 재사용 전략을 조정했다.

Step 1

smoke run으로 LGBM, FM, Deep&Wide, DeepFM, DLRM 코드 경로가 모두 돌아가는지 확인했다. 이 단계의 수치는 성능 결론에 쓰지 않고 실행 검증용으로만 봤다.

Step 2

hist28_mini, hist28_mid처럼 작은 catalog와 짧은 epoch로 후보 생성, feature matrix, neural training 비용을 확인했다.

Step 3

최신 run에서는 Airflow가 만든 ranker feature parquet를 먼저 재사용했다. ALS retrieval과 feature 생성을 매번 반복하면 로컬 Docker 메모리 한계에 걸리기 쉽기 때문이다.

Step 4

batch_size=8192는 너무 느렸고 65536은 PyTorch segfault가 나서, 최종 완료 run은 32768, torch_threads=1, CPU로 고정했다.

DeepFM 같은 neural ranker의 튜닝 방식 DeepFM만 따로 과하게 튜닝하면 비교가 흐려지므로, FM/Deep&Wide/DeepFM/DLRM은 공통 training budget을 맞췄다. 공통값은 epochs=3, lr=1e-3, weight_decay=1e-5, dropout=0.1, hidden_dims=128,64, batch_size=32768이고, DeepFM의 FM factor는 16이다. 이 설정은 “최적 DeepFM”이라기보다 같은 조건에서 DeepFM 구조가 추가 이득을 내는지 보는 비교 기준이다.
영역 이번 설정 이유 다음 튜닝 후보

layer 수 128,64 2-layer MLP positive label이 877개라 깊게 쌓으면 과적합 위험이 크다. 256,128,64, 64,32 비교. NDCG@100과 calibration을 함께 확인.
dropout / weight decay dropout=0.1, weight_decay=1e-5 positive label이 적어 neural ranker가 특정 popularity/source rule에 과하게 붙는 것을 줄이기 위한 기본 regularization. dropout=0.0/0.2, weight_decay=1e-4, early stopping.

loss Two-Tower는 in-batch cross entropy, ranker는 weighted BCE retrieval은 vector similarity 학습, re-rank는 후보별 positive 확률 학습으로 역할이 다르다. BPR, sampled softmax, LambdaRank-style neural loss.
LGBM LambdaRank n_estimators=120, learning_rate=0.05, num_leaves=31 적은 positive label에서도 tabular feature의 threshold rule을 안정적으로 학습한다. num_leaves, min_child_samples, lambdarank_truncation_level, candidate group 설계.
class imbalance pos_weight=185.32 positive rate 0.537%라 그대로 학습하면 대부분 negative로 예측하기 쉽다. hard negative mining, per-user negative sampling, candidate source별 weighting.
candidate set candidate_k=120, hybrid_extra=80, related_candidate_cap=30 ALS 후보에 popularity/recent/related 후보를 섞어 recall과 cold coverage를 보완했다. Two-Tower 후보를 hybrid에 추가하고 source flag를 분리.

우리 코드에서 가장 크게 반영된 것

  • shared candidate cache로 모델별 후보군 차이를 제거했다.
  • ranker 입력 feature를 표준화해 neural 모델 학습을 안정화했다.
  • positive imbalance를 pos_weight로 보정했다.
  • MLflow에 exp_*, core_* param/metric을 남겨 run 비교가 가능하게 했다.

아직 하지 않은 튜닝

  • DeepFM의 factor_dim 8/32/64 비교.
  • DeepFM/Deep&Wide의 layer width와 dropout 조합 grid.
  • weighted BCE 대신 listwise/pairwise ranking loss 적용.
  • candidate source별 hard negative를 따로 샘플링하는 방식.
