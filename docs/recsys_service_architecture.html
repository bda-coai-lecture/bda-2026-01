추천 서비스 구조를 한 번에 이해하기

브라우저에서 보이는 화면, Vercel에 올라간 프론트엔드, 로컬 Mac에서 도는 ML 백엔드, Cloudflare 터널, 캐시가 각각 어떤 역할을 하는지 연결해서 본다. 핵심은 "웹사이트가 떠 있다"와 "추천 모델이 응답한다"가 서로 다른 문제라는 점이다.

프론트엔드 Next.js on Vercel. 화면과 API proxy route를 담당한다.
ML 백엔드 FastAPI on local Mac. parquet, SQLite, 모델 파일을 읽어서 추천을 만든다.
연결 방식 Vercel route handler가 Cloudflare tunnel URL로 로컬 API를 호출한다.
장애 포인트 Mac, FastAPI, tunnel 중 하나가 꺼지면 추천 API가 실패할 수 있다.

2. 왜 이렇게 나눴나

화면은 가볍다

Vercel에 올리기 좋다

프론트엔드는 HTML, CSS, JavaScript 중심이라 배포가 쉽고 빠르다. 화면이 뜨는 것 자체는 추천 모델 파일이 없어도 가능하다.

모델은 무겁다

로컬 산출물 의존성이 크다

추천 서버는 parquet, SQLite, Python 패키지, 모델 파일을 읽는다. 이 구조는 Vercel serverless에 그대로 올리기보다 별도 Python 서버가 더 자연스럽다.

터널은 임시 다리다

데모에는 빠르지만 영구 인프라는 아니다

Cloudflare quick tunnel은 로컬 서버를 외부에 빨리 공개하기 좋다. 대신 프로세스가 꺼지면 주소가 죽거나 바뀔 수 있다.

3. 프론트엔드, 백엔드, 데이터 분리는 일반적인 구조인가

대부분의 서비스는 관심사를 나눠서 만든다

화면을 그리는 코드, 비즈니스 로직을 처리하는 서버, 데이터를 저장하고 읽는 계층은 요구사항이 서로 다르다. 그래서 한 파일이나 한 서버에 모두 넣기보다 역할별로 나누는 경우가 많다. 규모가 작을 때는 하나로 시작해도 되지만, 배포 주기와 장애 원인, 성능 병목이 달라지기 시작하면 분리하는 쪽이 운영하기 쉽다.

Frontend

사용자와 직접 만나는 부분

버튼, 입력창, 테이블, 로딩 상태, 에러 메시지를 담당한다. 사용자의 브라우저에서 실행되는 JavaScript와 Vercel 같은 프론트엔드 플랫폼에서 내려주는 HTML이 중심이다.

  • 사용성, 반응 속도, 화면 구성에 민감하다.
  • API가 느리거나 실패할 때 어떻게 보여줄지도 여기서 결정한다.
  • 모델 파일이나 원본 데이터 전체를 들고 있을 필요는 없다.
Backend

규칙과 계산을 처리하는 부분

요청을 받고, 권한을 확인하고, 데이터나 모델을 조회해서 응답을 만든다. 추천 서비스에서는 후보 생성, 랭킹, fallback, 캐시 제어 같은 로직이 여기에 들어간다.

  • Python, Node.js, Java, Go 등 목적에 맞는 런타임을 고른다.
  • 프론트엔드가 내부 구현을 몰라도 되도록 API 형태로 감싼다.
  • 장애가 나도 화면 전체가 아니라 API 단위로 문제를 좁힐 수 있다.
Data / Model

상태와 근거를 보관하는 부분

원본 이벤트, feature mart, 모델 artifact, metadata cache처럼 결과를 만드는 데 필요한 재료를 보관한다. 데이터는 크고 변환 비용이 있으므로 화면 코드와 같은 방식으로 배포하지 않는다.

  • DB, object storage, parquet, SQLite, model registry가 여기에 해당한다.
  • 백엔드는 필요한 데이터만 읽어 응답 형태로 바꿔준다.
  • 데이터 갱신 주기와 화면 배포 주기는 보통 다르다.
좋은 점

역할이 분리되면 바꾸기 쉽다

  • 화면 디자인만 바꿀 때 모델 서버를 건드리지 않아도 된다.
  • 모델을 새 버전으로 교체해도 프론트엔드 API 계약이 유지되면 화면 코드는 그대로 둘 수 있다.
  • 장애가 났을 때 "화면 문제인지, API 문제인지, 데이터 문제인지"를 빠르게 나눠 볼 수 있다.
주의할 점

네트워크 경계가 생긴다

  • 프론트엔드와 백엔드 사이에 HTTP 호출이 생기므로 지연과 실패 가능성이 생긴다.
  • API 계약이 깨지면 화면은 떠도 데이터 표시가 실패할 수 있다.
  • 환경변수, CORS, 인증, 캐시 같은 연결 설정을 관리해야 한다.

4. 요청 하나가 지나가는 길

Trending
1
브라우저가 /api/recsys/trending?limit=20 호출 사용자는 Trending 탭을 누르거나 새로고침한다.
2
Vercel route handler가 캐시를 먼저 확인 1시간 안에 같은 limit으로 본 적이 있으면 캐시 결과를 바로 줄 수 있다.
3
캐시가 없으면 로컬 FastAPI 호출 /api/trending에서 최근 activity growth 기반 결과를 만든다.
Similar
1
owner/repo를 입력한다 예: microsoft/markitdown
2
Vercel이 /api/repos/{owner}/{repo}/related로 변환 프론트엔드용 API 경로와 백엔드용 API 경로를 분리한다.
3
백엔드가 repo-to-repo co-occurrence 결과를 조회 같은 사용자 히스토리에서 함께 등장한 저장소 신호를 활용한다.
For User
1
actor_id 기준으로 개인화 추천 요청 사용자 히스토리를 기반으로 후보를 만들고 랭킹한다.
2
백엔드가 two-stage pipeline 실행 후보 생성 후보군에 점수를 붙이고, ranker로 최종 순서를 정한다.
3
개인화 결과는 CDN 캐시하지 않는다 대신 같은 서버 인스턴스 안에서는 짧게 메모리 캐시를 둔다.

5. Cloudflare 터널은 여기서 무엇을 하나

문제 상황

로컬 서버는 원래 외부에서 바로 못 들어온다

FastAPI는 내 Mac의 127.0.0.1:8001에서 실행된다. 127.0.0.1은 "이 컴퓨터 자신"이라는 뜻이라, Vercel 서버가 인터넷 밖에서 이 주소로 직접 접속할 수 없다. 집이나 사무실 네트워크 안의 로컬 주소는 공개 URL이 아니다.

해결 방식

터널이 공개 URL과 로컬 포트를 이어준다

cloudflared tunnel --url http://127.0.0.1:8001을 실행하면 Cloudflare가 https://...trycloudflare.com 형태의 공개 URL을 만들어준다. Vercel은 그 URL로 요청하고, Cloudflare tunnel 프로세스가 요청을 내 Mac의 FastAPI로 전달한다.

Tunnel path
1
Vercel이 tunnel URL로 요청 NEXT_PUBLIC_RECSYS_API_BASE_URL에 들어 있는 https://...trycloudflare.com 주소를 사용한다.
2
Cloudflare가 실행 중인 tunnel 프로세스를 찾는다 이 프로세스는 내 Mac에서 Cloudflare 쪽으로 연결을 열어 둔다. 그래서 외부에서 들어오는 요청을 받을 수 있다.
3
요청이 로컬 FastAPI로 전달된다 최종 목적지는 http://127.0.0.1:8001/api/...이다.
4
응답이 같은 길을 거꾸로 돌아간다 FastAPI 응답 → Cloudflare → Vercel route handler → 브라우저 순서로 결과가 도착한다.
장점

설정이 빠르다

방화벽, 공유기 포트포워딩, 고정 IP 없이 로컬 서버를 외부에서 테스트할 수 있다. 데모나 프로토타입 검증에 특히 편하다.

한계

Quick tunnel은 임시적이다

터널 프로세스가 꺼지면 URL이 죽는다. 다시 켜면 주소가 바뀔 수 있고, 그러면 Vercel 환경변수와 배포도 다시 맞춰야 한다.

운영 대안

고정 터널 또는 상시 서버

더 안정적으로 쓰려면 Cloudflare named tunnel을 쓰거나, FastAPI 자체를 Cloud Run, Fly.io, Render, VPS 같은 서버에 배포한다.

6. 현재 API 매핑

화면에서 호출하는 경로 Vercel이 호출하는 백엔드 경로 주요 입력 캐시 정책
/api/recsys/trending /api/trending limit CDN 1시간 fresh, 7일 stale. 같은 limit이면 캐시 적중률이 높다.
/api/recsys/related /api/repos/{owner}/{repo}/related owner, repo, limit CDN 1시간 fresh, 7일 stale. 자주 쓰는 저장소는 백엔드 다운에도 버틴다.
/api/recsys/personalized /api/users/{actor_id}/recommendations actor_id, limit CDN 캐시 없음. 서버 인스턴스 메모리에서 10분 fresh, 24시간 stale.

7. 캐시는 무엇을 해결하고, 무엇을 해결하지 못하나

해결하는 문제

잠깐의 백엔드 장애를 흡수한다

  • 한 번 성공한 Trending 결과를 다시 보여줄 수 있다.
  • 자주 보는 Similar repo 결과를 백엔드 없이도 내려줄 수 있다.
  • 강의 중 새로고침하거나 같은 예제를 반복할 때 안정적이다.
  • 백엔드가 느릴 때도 일부 요청은 빠르게 응답한다.
해결하지 못하는 문제

처음 보는 요청은 여전히 백엔드가 필요하다

  • 캐시에 없는 actor_id는 백엔드가 꺼져 있으면 실패한다.
  • 처음 검색하는 owner/repo도 백엔드가 필요하다.
  • Vercel serverless 인스턴스 메모리는 영구 저장소가 아니다.
  • Quick tunnel 자체가 죽으면 새 주소로 다시 배포하거나 env를 갱신해야 한다.

8. 장애를 볼 때의 사고 순서

1
화면 자체가 안 뜨는가? Vercel 배포, DNS, Next.js 빌드 문제를 먼저 본다. 이 경우 HTML/JS도 내려오지 않는다.
2
화면은 뜨는데 결과가 비어 있거나 API error가 나는가? 프론트엔드는 살아 있고, 추천 API 경로 또는 ML 백엔드 연결 문제일 가능성이 높다.
3
Vercel API가 502, 530, 1033 같은 에러를 내는가? 로컬 FastAPI, Cloudflare tunnel, env의 tunnel URL이 살아 있는지 확인한다.
4
Trending은 되는데 For User만 느린가? 개인화 추천은 모델과 feature artifact를 더 많이 읽으므로 cold start가 더 길 수 있다.

9. 실행과 검증 명령

Local ML backend

FastAPI 서버

추천 결과를 실제로 계산하는 Python 서버다.

  • cd /Users/kakao/bda-2
  • PYTHONPATH=src uv run uvicorn ghrec.local_api:app --host 127.0.0.1 --port 8001
  • curl http://127.0.0.1:8001/health
Public tunnel

Cloudflare tunnel

Vercel에서 로컬 FastAPI로 들어오는 통로다.

  • cloudflared tunnel --url http://127.0.0.1:8001
  • curl https://your-tunnel.trycloudflare.com/health
  • URL이 바뀌면 Vercel env와 재배포가 필요하다.

10. 운영 관점에서의 다음 단계

현재 개선

캐시로 데모 안정성 확보

반복 조회되는 Trending, Similar 결과는 백엔드가 잠깐 꺼져도 이전 응답을 보여준다. 가장 적은 변경으로 체감 안정성을 올리는 방법이다.

중간 단계

Named tunnel 또는 고정 URL

Quick tunnel 대신 고정된 tunnel을 쓰면 URL 변경 문제가 줄어든다. 로컬 서버 구조는 유지하면서 연결 안정성을 올릴 수 있다.

최종 단계

ML 백엔드 상시 배포

Python FastAPI를 Cloud Run, Fly.io, Render, VPS 같은 곳에 올리면 로컬 Mac 의존성이 사라진다. 모델 파일과 데이터 artifact 배포 전략이 함께 필요하다.

핵심 정리

이 서비스는 "Vercel에 올라간 웹 앱"과 "로컬에서 도는 추천 모델 서버"가 분리되어 있다. 그래서 웹사이트가 정상이어도 ML 백엔드나 터널이 꺼지면 추천 결과는 실패할 수 있다. 캐시는 이미 성공한 요청을 재사용해서 흔들림을 줄여주지만, 처음 보는 요청까지 완전히 대체하지는 못한다.
