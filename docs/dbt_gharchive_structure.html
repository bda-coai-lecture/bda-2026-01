GitHub Archive dbt 프로젝트 구조 설명

대상: 데이터 분석/엔지니어링 입문자와 주니어 실무자 / 프로젝트: dbt/gharchive_metrics

이 문서는 dbt가 무엇인지부터 시작해서, 이 repo의 dbt 프로젝트가 어떤 순서로 데이터를 읽고, 정제하고, 지표 테이블을 만들고, 테스트와 문서를 붙이는지 실무 관점에서 설명한다.

한 줄 정의 dbt는 SQL 파일들을 의존성 있는 데이터 모델로 관리하고, 실행 순서와 테스트와 문서화를 자동화하는 도구다.
이 프로젝트의 역할 GitHub Archive 일자별 shard를 incremental fact로 집계하고, Metabase dashboard에서 쓰기 좋은 metric mart를 만든다.
핵심 산출물 DAU/WAU, 이벤트 믹스, user lifecycle, active user retention heatmap, semantic metrics.

1. dbt가 해결하는 문제

데이터 분석을 하다 보면 같은 SQL을 여러 번 복사해서 쓰게 된다. 처음에는 괜찮지만, 테이블이 많아지고 dashboard가 늘어나면 다음 문제가 생긴다.

  • 어떤 SQL을 먼저 실행해야 하는지 사람이 기억해야 한다.
  • 컬럼 이름이나 집계 정의가 dashboard마다 조금씩 달라진다.
  • 원본 데이터가 바뀌었을 때 어떤 지표가 영향받는지 추적하기 어렵다.
  • 테이블이 맞게 만들어졌는지 매번 눈으로 확인하게 된다.
dbt는 이 문제를 SQL 파일 + 의존성 그래프 + 테스트 + 문서로 푼다. Python ETL처럼 데이터를 직접 들고 처리하기보다, warehouse 안에서 SQL을 실행해 모델 테이블을 만든다.

3. dbt에서 말하는 모델

dbt에서 모델(model)은 보통 models/ 아래에 있는 SQL 파일 하나를 뜻한다. 이 SQL 파일은 단순한 쿼리 저장소가 아니라, dbt DAG의 한 노드다. dbt는 ref()를 읽어서 모델 간 의존성을 만들고, 그 순서대로 SQL을 warehouse에서 실행한다.

SQL 파일

metrics_daily.sql처럼 하나의 select 문을 담는다. dbt는 이 select 결과를 view/table/incremental table로 materialize한다.

DAG 노드

ref('stg_user_repo_activity')를 쓰면 해당 모델이 staging 모델 뒤에 실행되어야 한다는 의존성이 생긴다.

계약 단위

모델은 downstream 사용자가 기대하는 컬럼과 grain을 제공한다. 예: metrics_weekly는 주별 한 행이다.

테스트/문서 단위

schema.yml에서 모델 설명, 컬럼 설명, not_null, unique 같은 테스트를 붙인다.

Materialization: 모델이 실제로 저장되는 방식

방식 의미 이 프로젝트에서 쓰는 곳
view 쿼리 정의만 저장한다. 조회할 때 계산된다. 가볍고 staging에 적합하다. stg_user_repo_activity
table 쿼리 결과를 물리 테이블로 저장한다. dashboard가 반복 조회하는 mart에 적합하다. metrics_daily, metrics_weekly
incremental 매번 전체를 다시 만들지 않고, 지정한 범위나 변경분만 반영한다. 대용량 raw를 다룰 때 필요하다. fact_user_repo_activity
이 프로젝트의 핵심은 fact_user_repo_activity를 incremental 모델로 두고, 그 위의 staging과 marts는 ref()로 이어지는 dbt-owned DAG 안에 넣는 것이다.

6. dbt_project.yml: 프로젝트 설정의 입구

dbt_project.yml은 이 프로젝트의 이름, 모델 경로, 테스트 경로, materialization 기본값을 정한다. 이 파일을 보면 dbt가 어디에서 SQL을 찾고, 어떤 모델을 view/table로 만들지 알 수 있다. 

name: gharchive_metrics
profile: gharchive_metrics

model-paths: ["models"]
test-paths: ["tests"]

models:
  gharchive_metrics:
    raw:
      +materialized: view
      +tags: ["lineage_reference"]
    staging:
      +materialized: view
    facts:
      +materialized: incremental
    marts:
      +materialized: table

여기서는 facts는 incremental, staging은 view, marts는 table로 만든다. fact는 일자별 partition을 overwrite하고, staging은 가벼운 정제 계층이라 view로 충분하며, marts는 dashboard가 반복 조회하므로 table로 만들어 둔다.

7. Fact와 staging 계층

Fact: dbt가 소유하는 incremental 적재 경계

models/facts/fact_user_repo_activity.sql는 GitHub Archive의 일자별 shard를 읽어서 activity_date, user_id, repo_id, action grain의 fact table로 집계한다. 이 모델은 dbt가 소유하는 incremental table이다.

Actor 기준: 현재 user_id는 GitHub Archive의 actor.id다. 이름은 dashboard 호환성 때문에 user_id로 남아 있지만, 의미상으로는 actor_id에 가깝다. 예를 들어 A 유저가 B organization 아래에서 이벤트를 발생시키면 이 fact에는 B가 아니라 A가 active actor로 기록된다. GitHub Archive의 org context는 아직 저장하지 않는다. 
{{
  config(
    materialized='incremental',
    partition_by={"field": "activity_date", "data_type": "date", "granularity": "day"},
    cluster_by=["user_id", "repo_id", "action"],
    incremental_strategy='insert_overwrite'
  )
}}

GitHub Archive는 날짜별 테이블이므로 전체 raw를 매번 스캔할 필요가 없다. raw_start_date~raw_end_date 범위의 shard만 읽고, 해당 날짜 partition만 overwrite한다.

기존 mart.fact_user_repo_activity는 현재 BigQuery 메타데이터 기준 unpartitioned table이다. 따라서 insert_overwrite incremental 전략을 실제 적용하려면 한 번은 partitioned table로 마이그레이션해야 한다. 이후부터는 일자별 shard와 해당 날짜 partition만 다루면 된다.

Staging: 분석하기 좋은 기본 형태로 정리

stg_user_repo_activity.sql는 dbt fact 모델을 읽어서 타입을 확정하고, 주/월 분석에 필요한 날짜 컬럼을 만든다. 

select
  cast(user_id as int64) as user_id,
  cast(repo_id as int64) as repo_id,
  cast(action as string) as action,
  cast(event_count as int64) as event_count,
  cast(activity_date as date) as activity_date,
  date_trunc(cast(activity_date as date), week(monday)) as week_start,
  date_trunc(cast(activity_date as date), month) as month_start
from {{ ref('fact_user_repo_activity') }}
where user_id is not null
  and repo_id is not null
  and action is not null
  and event_count is not null
  and activity_date is not null
staging의 목적은 복잡한 비즈니스 지표를 만들기보다, 모든 downstream 모델이 같은 깨끗한 입력을 쓰게 하는 것이다.

8. Mart 계층: dashboard가 바로 쓰는 지표 테이블

models/marts/platform 아래 모델들은 Metabase dashboard나 분석 노트북에서 바로 쓰기 좋게 만든 결과 테이블이다. 이름이 metrics_로 시작하는 이유도 "지표 mart"라는 의미다.

이 문서와 dashboard에서 쓰는 active_users, weekly_active_users, WAU, MAU는 모두 GitHub Archive event actor 기준이다. Organization 자체의 활성도를 보려면 별도 org_id, org_login 컬럼을 fact에 추가해야 한다.

모델 역할 대표 컬럼
metrics_daily 일별 전체 활동 지표. DAU, active repos, events, 이벤트 타입별 count. activity_date, active_users, total_events
metrics_weekly 주별 활동 지표. WAU와 weekly events를 본다. week_start, weekly_active_users
metrics_event_type_daily 이벤트 타입별 일별 활동. Push, Watch, Fork 등의 구성 변화를 본다. activity_date, action, active_users
metrics_user_segments 유저를 활동 빈도 기준으로 one-day, repeat, regular, power로 나눈다. user_segment, users, avg_active_days

9. Retention 모델을 예시로 이해하기

이번에 중요한 모델은 metrics_cohort_retention_weekly_heatmap이다. 여기서 "cohort"라는 단어가 남아 있지만, 의미는 first-seen 신규 유저 cohort가 아니다. 각 주의 전체 WAU를 기준 집단으로 잡는다. 

with user_week as (
  select distinct
    user_id,
    week_start
  from {{ ref('stg_user_repo_activity') }}
),

cohort_activity as (
  select
    cohort_week.week_start as cohort_week,
    activity_week.week_start,
    date_diff(activity_week.week_start, cohort_week.week_start, week) as weeks_since,
    cohort_week.user_id
  from user_week as cohort_week
  join user_week as activity_week
    on cohort_week.user_id = activity_week.user_id
    and activity_week.week_start >= cohort_week.week_start
    and activity_week.week_start <= date_add(cohort_week.week_start, interval 12 week)
)

이 SQL은 같은 유저가 기준 주에도 활동했고, 이후 주에도 활동했는지 자기 자신과 join해서 확인한다. 마지막 출력은 dashboard에서 보기 쉽게 wide table로 만든다. 

select
  cohort_week as week_start,
  max(cohort_users) as active_users,
  max(if(weeks_since = 1, retention_rate, null)) as w1,
  max(if(weeks_since = 2, retention_rate, null)) as w2,
  ...
  max(if(weeks_since = 12, retention_rate, null)) as w12
from retention
group by cohort_week

숫자로 보는 해석 예시

week_start active_users w1 w2 해석
2026-04-06 100,000 0.42 0.31 2026-04-06 주 활동자 100,000명 중 42,000명이 다음 주에도, 31,000명이 2주 뒤에도 활동했다.

Retention heatmap이 보여주는 형태

week_start active_users w1 w2 w3 w4 w5 w6
2026-03-23 1,980,000 52% 41% 35% 29% 24% 21%
2026-03-30 2,050,000 50% 39% 34% 28% 23% null
2026-04-06 2,120,000 47% 37% 31% 26% null null
2026-04-13 2,180,000 45% 33% 27% null null null

색이 진할수록 같은 기준 주의 active users가 이후 기간에도 다시 활동한 비율이 높다는 뜻이다. 오른쪽 아래 null은 아직 미래 기간이라 관측되지 않은 값이다.

최신 주의 w1 이후 값이 null이면 "아직 미래 데이터가 없어서 모른다"는 뜻이다. 0% retention으로 해석하면 안 된다.

10. 테스트: 데이터가 맞는지 자동으로 확인하기

dbt test는 "테이블이 만들어졌다"에서 끝내지 않고, 만들어진 데이터가 기본 조건을 만족하는지 확인한다.

테스트 검증하는 것
not_null 핵심 키나 지표 컬럼이 비어 있지 않은지 확인한다.
unique metrics_daily.activity_date, metrics_weekly.week_start처럼 한 기간에 한 행만 있는지 확인한다.
metrics_cohort_retention_weekly_heatmap_rates w1~w12 값이 0과 1 사이인지 확인한다.
metrics_cohort_retention_weekly_heatmap_active_users_match_wau heatmap의 active_users가 metrics_weekly.weekly_active_users와 같은지 확인한다.
assert_user_lifecycle_weekly_balances retention/churn/lifecycle 계산의 사용자 수 균형이 맞는지 확인한다.

품질 검증이 보는 네 가지 축

Completeness

핵심 기간과 지표 컬럼이 비어 있지 않은지 확인한다. 예: week_start, active_users.

Uniqueness

일별/주별 mart가 한 기간에 한 행만 가지는지 확인한다. 중복 행은 dashboard 집계를 망친다.

Range

retention rate처럼 비율인 값이 0~1 범위를 벗어나지 않는지 확인한다.

Reconciliation

heatmap의 active_users가 독립적으로 계산한 WAU/MAU와 일치하는지 대조한다.

12. Semantic Layer는 왜 따로 있나

Semantic Layer는 mart table 위에 한 번 더 올라가는 지표 정의 계층이다. mart가 "조회 가능한 테이블"을 만드는 계층이라면, semantic layer는 "이 테이블에서 어떤 차원으로 어떤 지표를 계산할 수 있는지"를 선언한다. 같은 지표를 여러 dashboard, notebook, API, MetricFlow query에서 쓰더라도 정의가 갈라지지 않게 하는 목적이다.

이 프로젝트의 semantic layer는 models/semantic/platform_activity.yml에 있다. 기반 모델은 stg_user_repo_activity이고, 그 위에 entity, dimension, measure, metric을 정의한다.

Semantic Model 어떤 dbt 모델을 분석 가능한 의미 계층으로 볼지 정한다. 여기서는 stg_user_repo_activity를 사용한다.
Entity 분석 대상의 식별자다. 예: user, repo, activity.
Dimension 지표를 자르는 기준이다. 예: activity_date, week_start, action.
Measure 원천 컬럼에 적용할 집계 규칙이다. 예: sum(event_count), count_distinct(user_id).
Metric 외부에서 호출할 지표 이름이다. 예: active_users, total_events.

이 프로젝트의 semantic model 예시

semantic_models:
  - name: user_repo_activity
    model: ref('stg_user_repo_activity')
    defaults:
      agg_time_dimension: activity_date
    entities:
      - name: user
        type: foreign
        expr: user_id
      - name: repo
        type: foreign
        expr: repo_id
    dimensions:
      - name: activity_date
        type: time
      - name: action
        type: categorical
    measures:
      - name: active_users
        agg: count_distinct
        expr: user_id

위 정의가 있으면 active_users는 "유저 수를 아무렇게나 세는 값"이 아니라 user_id의 distinct count라는 계약이 된다. total_events도 event_count의 sum으로 정의된다.

total_events active_users active_repos push_events watch_events fork_events pull_request_events issue_events

14. 실무 기준 체크포인트

  • 지표 모델은 이름보다 정의가 먼저다. 특히 retention은 denominator와 관측 기간을 명시해야 한다.
  • 일자별 raw shard는 incremental fact로 관리하는 게 자연스럽다. 전체 raw scan을 기본값으로 두면 안 된다.
  • fact, staging, marts의 책임을 섞지 않는다. 적재 경계, 정제, 집계, serving 목적을 분리해야 변경 영향이 작다.
  • ref() 기반 의존성을 유지해야 lineage, 선택 실행, 영향 분석이 가능하다.
  • dashboard에 노출되는 mart는 최소한 기간 키 unique, 핵심 지표 not null, 비율 범위, reconciliation 테스트를 갖춰야 한다.
  • 문서화는 부가 작업이 아니다. 지표 정의, 예외 처리, null 해석은 dashboard 사용자에게 전달되는 계약이다.
