1. Local Machine: 원천 데이터와 실행 환경

모든 데이터 플랫폼은 먼저 "어디에 원본이 있는가"에서 시작한다. 여기서는 클라우드 데이터레이크를 크게 만들기 전에, 로컬 머신에 있는 parquet 파일과 SQLite cache가 작은 원천 저장소 역할을 한다.

로컬 작업공간 안에서 원천 파일, metadata cache, 추천 산출물이 분리되어 있고 Airflow 컨테이너는 Docker volume mount로 이 파일들을 읽는다.

무엇을 보관하나
  • data/daily_agg/*.parquet: GitHub Archive 일별 집계
  • data/repo_metadata.db: repo metadata cache
  • data/models/week6: 추천 모델/평가 산출물
현재 상태

로컬 parquet는 2026-01-01부터 2026-05-16까지 136일치가 있다. 다만 매일 전체를 다시 처리하면 느리고 비용도 커지기 때문에, 운영 DAG는 최근 35일만 처리하도록 제한한다.

왜 중요한가

원천 위치가 분명해야 같은 데이터를 여러 번 다른 방식으로 복사하지 않는다. BI 지표와 추천 실험이 같은 parquet에서 출발하기 때문에 결과를 서로 비교할 수 있다.

왜 이 방식을 쓰나처음부터 복잡한 cloud data lake를 만들기보다, 로컬 parquet로 빠르게 실험하고 같은 파일을 Airflow, dbt, 추천 코드가 공유하게 만든다.
장점세팅이 단순하고 재실행이 쉽다. 파일이 날짜별로 나뉘어 있어 필요한 기간만 읽을 수 있다.
한계로컬 머신에 의존하므로 팀 단위 공유와 권한 관리에는 약하다. 데이터가 커지면 저장공간과 처리 시간이 문제가 된다.
다음 단계운영 규모가 커지면 GCS 같은 object storage를 원천 저장소로 두고, BigQuery external table이나 적재 테이블로 연결한다.
parquet행과 열이 많은 분석 데이터를 빠르게 읽기 좋은 파일 포맷이다. CSV보다 크기가 작고 필요한 컬럼만 읽기 쉽다.
metadata cacheGitHub API를 매번 호출하지 않도록 repo의 부가정보를 로컬 DB에 저장해두는 장치다.
window파이프라인이 처리할 날짜 범위다. 이번 구조에서는 최신 35일이 운영 처리 단위다.

2. Airflow: 사람이 하던 실행 순서의 자동화

데이터 작업은 보통 "파일 확인 → 변환 실행 → 테이블 적재 → 대시보드 확인"처럼 순서가 있다. Airflow는 이 순서를 코드로 고정하고, 매일 같은 시간에 실행하며, 실패했을 때 어느 단계에서 멈췄는지 보여준다.

현재 compose는 LocalExecutor라 scheduler가 metadata DB를 보고 실행 가능한 task instance를 고른 뒤 같은 Airflow 컨테이너 안의 로컬 프로세스로 실행한다. Redis와 Celery worker를 쓰지 않기 때문에 단순하지만, 동시에 오래 돌릴 수 있는 작업 수는 한 머신의 CPU와 메모리에 묶인다.

회사에서는 Airflow가 직접 무거운 계산을 오래 들고 있기보다, Kubernetes pod나 autoscaled worker pool 같은 실행 계층에 task를 넘긴다. 서버리스 job 실행기를 쓰는 경우도 있지만, 핵심은 Airflow가 “언제 무엇을 실행할지”를 조율하고 실제 compute는 별도의 worker 계층이 담당한다는 점이다.

Scheduler가 하는 일

scheduler는 cron처럼 단순히 시간을 맞추는 프로세스가 아니다. DAG 파일을 해석한 결과와 metadata DB의 상태를 보고, 어떤 DAG run을 만들지, 어떤 task가 dependency를 만족했는지, pool과 concurrency 제한 안에서 무엇을 queue에 넣을지를 계속 결정한다.

확장할 때 어려운 이유

task 수가 많아지면 scheduler는 수천 개 task instance의 상태를 계속 평가해야 한다. 여기서 DAG parsing, metadata DB 부하, executor queue, worker slot, retry/backfill이 모두 얽힌다. 그래서 scheduler 자체를 무겁게 만들기보다 task 실행은 worker 계층으로 넘기고, metadata DB와 queue를 안정적으로 운영해야 한다.

실행 방어선

plan_*_sync 태스크가 날짜 범위와 max_days=35를 먼저 확인한다. 실수로 너무 긴 기간을 처리하거나, 이미 있는 추천 feature를 불필요하게 다시 만들지 않도록 막는다.

이 프로젝트의 DAG
  • gharchive_repo_metadata_refresh: repo metadata 갱신
  • gharchive_platform_metrics: metric marts 직접 적재
  • gharchive_dbt_metrics: fact 적재 후 dbt build
  • gharchive_recsys_features: 추천 mart와 feature cache 생성
LocalExecutor에서 운영 확장으로

로컬에서는 scheduler와 task 실행이 한 머신 안에 있다. 운영에서는 CeleryExecutor로 worker pool을 늘리거나, KubernetesExecutor/KubernetesPodOperator로 task마다 pod를 띄운다. 이때 scheduler는 “실행 명령을 내리는 두뇌”이고, 실제 CPU 작업은 worker/pod/job이 담당한다.

왜 중요한가

Airflow는 계산 엔진이 아니라 조율자다. 팀에서 누가 실행해도 같은 순서로 돌고, 로그와 retry가 남기 때문에 "내 컴퓨터에서는 됐는데" 문제를 줄인다.

왜 이 도구를 쓰나데이터 작업은 순서와 시간이 중요하다. Airflow는 여러 작업을 DAG로 연결하고, 매일 같은 시간에 실행되도록 만든다.
장점UI에서 성공/실패를 확인할 수 있고, 각 task의 로그가 남는다. retry와 timeout을 설정해 운영 사고를 줄일 수 있다.
한계Airflow 자체가 데이터를 빠르게 처리해주는 엔진은 아니다. task 안에서 실행되는 Python, SQL, dbt, BigQuery 작업이 실제 계산을 한다.
운영 확장회사에서는 KubernetesExecutor, CeleryExecutor, K8sPodOperator처럼 task 실행 계층을 분리해 worker를 탄력적으로 늘린다. 서버리스 job 호출은 별도 선택지로 볼 수 있다.
DAG작업들의 순서도다. A가 끝나야 B가 실행되는 관계를 코드로 표현한다.
taskDAG 안의 한 단계다. 예를 들어 날짜 확인, fact 적재, dbt build가 각각 task가 된다.
retry / log실패한 작업을 다시 시도하고, 실패 원인을 나중에 확인할 수 있게 남기는 운영 장치다.
scheduler시간표, 의존성, 과거 실행 상태, 동시성 제한을 보고 다음에 실행할 task instance를 결정하는 Airflow의 핵심 프로세스다.
executorscheduler가 고른 task를 실제 어디에 보낼지 정하는 실행 방식이다. Local, Celery, Kubernetes executor가 대표적이다.
workertask 코드를 실제로 실행하는 프로세스나 pod다. 확장성은 보통 scheduler보다 worker 계층을 얼마나 탄력적으로 늘릴 수 있느냐에서 나온다.

3. BigQuery: 공유 Warehouse

로컬 파일만으로도 분석은 가능하지만, 여러 도구가 같은 데이터를 조회하려면 공용 저장소가 필요하다. BigQuery는 여기서 Metabase와 dbt가 함께 바라보는 기준 테이블 저장소 역할을 한다.

큰 fact는 상세 분석과 dbt source가 되고, 대시보드는 미리 작게 만든 metric mart를 조회해 비용과 지연을 줄인다.

BigQuery는 데이터를 Colossus에 columnar 형태로 저장하고, SQL 실행 시 필요한 컬럼과 파티션만 읽으려 한다. 그래서 WHERE activity_date BETWEEN ...처럼 partition key를 조건에 넣고, repo나 event type으로 자주 필터링한다면 clustering을 맞추는 것이 비용과 속도에 직접 영향을 준다.

저장되는 테이블
  • fact_user_repo_activity: user-repo-action fact
  • metrics_daily, metrics_weekly
  • metrics_retention_summary
  • metrics_agent_trendy_repos
Partition / Clustering

현재 sandbox 적재는 만료 정책 제약 때문에 fact를 단순 테이블로 두지만, 운영 fact는 보통 activity_date로 partition을 나눈다. 자주 함께 필터링되는 repo_id, action_type, user_id 같은 컬럼은 clustering 후보가 된다.

현재 적재 결과

최근 수동 갱신에서 35일치 fact 46,097,702 rows가 적재됐고, metric mart 8개가 Metabase 조회용으로 갱신됐다. 지표 테이블은 작게 만들어 대시보드 조회 비용을 줄인다.

왜 이 도구를 쓰나로컬 파일은 개인 작업에는 충분하지만, BI와 dbt가 함께 보려면 SQL로 접근 가능한 공용 warehouse가 필요하다.
장점storage와 compute가 분리되어 있어 서버를 직접 관리하지 않고도 큰 데이터를 SQL로 조회할 수 있다. Metabase, dbt, Python client가 모두 같은 테이블을 바라본다.
한계스캔한 데이터량에 따라 비용이 발생한다. partition filter 없이 큰 fact를 조회하거나 필요한 컬럼보다 많이 읽으면 대시보드 비용이 쉽게 커진다.
이 프로젝트에서bda-coai.mart dataset이 fact, metric mart, trend table의 기준 저장소다.
partition테이블을 날짜 같은 기준으로 물리적으로 나눠 읽을 범위를 줄이는 방식이다. 시간 단위 로그/이벤트 fact에서 가장 먼저 보는 최적화다.
clustering파티션 안에서 특정 컬럼 값이 비슷한 row를 가까이 두는 방식이다. repo별, action별, user별 필터가 많을 때 효과가 난다.
ColossusBigQuery 밑에서 데이터를 저장하는 Google의 분산 저장 계층이다. BigQuery는 이 저장소 위에서 필요한 컬럼과 block을 병렬로 읽는다.
fact table가장 자세한 사건 기록에 가까운 테이블이다. 여기서는 날짜, 사용자, repo, action 단위의 활동 집계다.
mart table대시보드나 분석 목적에 맞게 미리 요약한 테이블이다. DAU, WAU, retention 등이 여기에 해당한다.
cost control큰 fact를 매번 스캔하지 않고 작은 mart를 조회하면 속도와 비용을 함께 줄일 수 있다.

4. dbt: 지표 정의를 코드로 고정

"DAU를 어떻게 계산할 것인가" 같은 규칙은 생각보다 쉽게 어긋난다. dbt는 지표를 만드는 SQL과 테스트를 프로젝트 안에 남겨서, 지표 정의가 사람의 기억이 아니라 코드로 관리되게 한다.

dbt는 source에서 staging, mart, semantic metrics로 이어지는 lineage를 만든다. 그래서 어떤 지표가 어떤 원천에서 계산됐는지 추적할 수 있다.

모델 계층
  • stg_user_repo_activity: source 정리
  • metrics_daily: DAU, active repo, events
  • metrics_event_type_daily: action별 지표
  • metrics_retention_weekly: cohort retention
Semantic Metrics

active_users, active_repos, total_events, push_events, watch_events 같은 이름을 YAML로 고정한다.

왜 중요한가

dbt의 역할은 SQL 실행만이 아니다. 지표를 재사용 가능한 계약으로 만들고, 모델과 테스트를 코드 리뷰 대상으로 만든다.

왜 이 도구를 쓰나SQL이 여러 파일과 노트북에 흩어지면 같은 지표도 다르게 계산된다. dbt는 변환 SQL을 프로젝트 구조 안에 묶는다.
장점source, staging, mart 계층을 명확히 나눌 수 있고, 테스트와 문서화를 함께 둘 수 있다.
한계dbt는 batch SQL transform에 강하지만, API 호출이나 복잡한 Python feature engineering은 별도 스크립트가 더 편할 수 있다.
이 프로젝트에서핵심 platform metric은 dbt로 옮겼고, metadata enrichment가 필요한 trend table은 아직 Python이 담당한다.
staging model원천 테이블을 바로 쓰지 않고, 컬럼명과 타입을 정리해 다음 단계가 읽기 좋게 만든다.
metric model비즈니스 질문에 바로 답할 수 있게 날짜별, 주차별, cohort별로 집계한 모델이다.
semantic layer같은 지표명을 여러 도구에서 일관되게 쓰기 위한 이름과 계산 방식의 사전이다.

5. Metabase: 지표를 소비하는 화면

데이터가 테이블에만 있으면 SQL을 아는 사람만 볼 수 있다. Metabase는 만들어진 지표를 차트와 대시보드로 보여줘서, 데이터 파이프라인의 결과가 실제로 확인 가능한 화면이 되게 한다.

Metabase는 계산을 소유하기보다 결과를 보여주는 계층이다. 계산 규칙은 BigQuery/dbt에 두고, Metabase는 card와 dashboard로 소비한다.

scripts/setup_metabase_dashboard.py는 Metabase REST API로 database 연결, collection, card, dashboard 배치를 자동 구성한다. MCP/agent는 사람이 브라우저에서 반복 클릭하던 일을 줄이기 위해 대시보드 상태와 요청을 읽고, 어떤 SQL/card/layout을 바꿀지 제안하거나 API 호출 계획으로 바꾸는 인터페이스로 설명할 수 있다.

대시보드
  • GitHub Archive 기초 지표
  • AI Agent 트렌디 레포
  • OSS Signal 운영 대시보드
조회 방식

Metabase는 raw parquet나 큰 fact를 직접 보지 않고, BigQuery의 작은 metric mart와 trend table을 조회한다. 그래서 대시보드는 빠르게 열리고, 계산 규칙은 dbt/BigQuery 쪽에 남는다.

API 자동화

setup_metabase_dashboard.py는 로컬 Metabase가 뜰 때까지 기다린 뒤 admin session을 만들고, BigQuery database, collection, SQL card, dashboard layout을 API로 생성한다. 대시보드도 코드처럼 다시 만들 수 있게 하는 장치다.

MCP/agent가 붙는 지점

MCP는 BI 화면을 사람이 직접 눌러 수정하는 대신, agent가 “현재 어떤 card가 있고 어떤 SQL을 쓰는지”를 맥락으로 받아 변경안을 만들게 하는 연결 계층으로 다룰 수 있다. 핵심은 대시보드가 단순 산출물이 아니라, API로 읽고 쓸 수 있는 운영 객체가 된다는 점이다.

좋은 책임 분리

지표 계산은 dbt/BigQuery가 소유하고, Metabase는 시각화와 배치를 소유한다. agent나 MCP가 개입하더라도 지표 정의를 카드 안에 복붙해서 늘리지 않고, 검증된 mart를 조회하는 SQL과 dashboard UX를 조정하는 쪽에 집중한다.

왜 중요한가

분석 결과가 노트북 안에 있으면 공유가 어렵다. BI 화면으로 바뀌면 SQL을 직접 실행하지 않아도 같은 숫자를 볼 수 있고, API/MCP 자동화가 있으면 대시보드도 재현 가능한 운영 자산으로 관리할 수 있다.

왜 이 도구를 쓰나분석 결과를 SQL이나 노트북에만 두면 설명은 가능하지만 반복 확인이 어렵다. Metabase는 지표를 화면으로 공유한다.
장점카드와 대시보드를 빠르게 만들 수 있고, SQL을 몰라도 결과를 볼 수 있다. REST API가 있어 dashboard provisioning과 반복 수정도 자동화할 수 있다.
한계계산 규칙을 Metabase 카드마다 직접 넣으면 지표가 다시 흩어진다. MCP/agent를 붙여도 원천 지표 계약은 dbt/BigQuery 쪽에 남겨야 한다.
이 프로젝트에서기초 지표, AI Agent 트렌디 레포, OSS Signal 운영 대시보드가 BigQuery mart를 조회하고, API 스크립트가 카드와 레이아웃을 구성한다.
dashboard여러 지표와 차트를 한 화면에 묶은 관측 화면이다.
cardMetabase 안에서 하나의 SQL 질문이나 차트 단위를 뜻한다.
serving layer가공된 데이터를 실제 사용자가 보는 형태로 제공하는 마지막 계층이다.
Metabase API브라우저 클릭 대신 database, collection, card, dashboard를 HTTP 요청으로 만들고 수정하게 해주는 운영 인터페이스다.
MCPagent가 BI 도구의 상태와 사용자의 변경 의도를 도구 호출 가능한 맥락으로 다룰 수 있게 해주는 연결 방식이다.
provisioning대시보드를 수작업 결과물이 아니라 스크립트로 재생성 가능한 설정으로 관리하는 방식이다.

6. Recommendation Pipeline: 추천 실험의 배치화

추천 모델은 단순히 모델을 학습하는 것만으로 끝나지 않는다. 어떤 기간을 과거로 볼지, 어떤 기간을 정답으로 볼지, 후보를 어떻게 만들지, 어떤 feature로 ranker를 학습할지를 고정해야 실험을 다시 재현할 수 있다.

추천 파이프라인은 기간을 나누고, 후보 생성과 ranking feature 생성을 분리한다. 이 구조가 있어야 실험 결과를 다시 만들고 비교할 수 있다.

기간 계약
  • history: 2026-03-22 ~ 2026-05-02
  • rank-label: 2026-05-03 ~ 2026-05-09
  • test: 2026-05-10 ~ 2026-05-16
산출물

추천 mart 5종과 ranker_features_airflow_20260516.parquet가 생성된다. 최신 feature cache는 75개 feature와 163,400 rank rows를 가진다.

왜 중요한가

추천도 노트북 실험에서 끝나지 않는다. mart, retrieval, feature cache로 나눠야 결과를 비교하고, 다음 실험에서 같은 조건을 다시 만들 수 있다.

왜 이렇게 나누나추천은 모델 하나보다 데이터 준비가 더 중요하다. history, label, test 기간을 나누고 후보 생성과 ranking을 분리해야 평가가 가능하다.
장점mart와 feature cache가 남아 실험을 재현할 수 있다. retrieval과 ranker를 따로 개선할 수 있다.
한계현재는 batch 실험 구조라 실시간 추천이나 online feature serving은 없다. repo metadata도 완전한 point-in-time snapshot은 아니다.
이 프로젝트에서ALS retrieval, related candidate, 75개 ranker feature를 사용해 LGBM ranker 학습 데이터로 연결한다.
retrieval전체 repo를 다 점수화하지 않고, 먼저 후보를 좁히는 단계다. 여기서는 ALS, popular, related 후보를 함께 쓴다.
rankerretrieval이 만든 후보를 여러 feature로 다시 정렬하는 모델이다.
feature cache모델 학습에 필요한 입력값을 파일로 고정해, 같은 실험을 다시 돌릴 수 있게 만든 결과물이다.
