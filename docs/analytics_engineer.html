Grain을 쉽게 이해하기

Grain은 "이 테이블의 한 행이 정확히 무엇을 뜻하는가"에 대한 약속이다. 컬럼 이름 목록이 아니라 데이터의 해상도다.

Raw event grain

GitHub Archive 원천은 이벤트 1건이 한 행이다. 같은 유저가 같은 repo에서 하루에 20번 push하면 20행이 생길 수 있다.

Fact grain

fact_user_repo_activity는 activity_date, user_id, repo_id, action 조합이 한 행이다. 하루에 한 actor가 한 repo에서 한 event type을 몇 번 만들었는지가 event_count로 접힌다.

Mart grain

metrics_daily는 activity_date가 한 행이고, metrics_weekly는 week_start가 한 행이다. dashboard가 바로 읽기 좋은 단위로 더 접힌다.

사진으로 비유하면 grain은 픽셀 크기다. 픽셀이 너무 크면 세부 행동을 잃고, 너무 작으면 매번 중복 제거와 비용 문제가 생긴다. Analytics engineer는 질문에 맞는 해상도를 고른다.

질문이 grain을 결정한다

"오늘 활성 유저는 몇 명인가?" 필요한 기준은 activity_date + user_id다. event row를 그대로 count(*)하면 이벤트 수가 되지 유저 수가 아니다.
"어떤 repo가 많이 쓰였나?" activity_date + repo_id 또는 week_start + repo_id 단위가 필요하다. user 단위만 남기면 repo별 활동을 복원할 수 없다.
"유저가 다음 주에도 남았나?" user_id + cohort_week + active_week가 필요하다. 최신 주의 null은 0%가 아니라 아직 미래를 모른다는 뜻일 수 있다.
"추천 모델 interaction은 무엇인가?" window_end_date + actor_id + repo_id가 필요하다. 학습 시점 이후 feature가 섞이면 leakage가 생긴다.
"AI agent trend repo는 무엇인가?" seed repo, keyword 후보, recent window, baseline window, weighted score, seed affinity가 모두 정의되어야 agent가 같은 후보를 찾는다.
"조직 활동도 포함할까?" 현재 user_id는 GitHub actor.id다. organization context는 별도 grain과 컬럼을 추가하지 않으면 active user에 포함되지 않는다.

Grain이 틀리면 생기는 문제

DAU 과대 집계event row를 세면 활동 유저가 아니라 활동 이벤트 수가 된다.
Retention 오해first-seen cohort와 전주 대비 lifecycle retention을 섞으면 같은 retention이라는 이름으로 다른 숫자가 나온다.
Dashboard 불일치대시보드마다 raw를 직접 집계하면 bot 포함, timezone, 기간 기준이 갈라진다.
ML leakagefeature snapshot 기준이 없으면 테스트 기간 이후 정보를 학습 데이터에 넣을 수 있다.
AI 환각semantic layer 없이 agent가 테이블을 추측하면 그럴듯하지만 틀린 SQL을 만들기 쉽다.

Analytics Engineer가 관리하는 데이터 계약

지표는 이름보다 정의가 중요하다

`active_users`는 GitHub Archive의 `actor.id` 기준인지, organization context를 포함하는지에 따라 다른 숫자가 된다. retention도 first-seen cohort, active-user retention, period-over-period lifecycle retention이 서로 다르다.

  • active_users: distinct actor 기준
  • retention_weekly: 첫 관측 cohort 기준
  • user_lifecycle_weekly: 전주 대비 유지/복귀/이탈 기준

추천 실험도 데이터 계약이 필요하다

ALS, LGBM, Two-Tower 자체는 ML 영역에 가깝지만, 모델이 읽는 feature와 split의 기준은 analytics engineering과 강하게 연결된다.

  • window_end_date로 interaction 기준 시점 고정
  • snapshot_date로 repo feature 기준 시점 명시
  • experiment_split_mart로 train/test 흔들림 방지

Metric은 계산식이 아니라 계약이다

좋은 지표 정의에는 이름, grain, 필터, 시간 기준, 소유 테이블, 테스트가 같이 있어야 한다. 그래야 사람, BI, ML, AI agent가 같은 질문에 같은 답을 낸다.

Metric계약에 포함할 것이 repo에서의 기준
active_users누구를 유저로 볼지, 기간 grain은 무엇인지, 중복 제거 기준은 무엇인지actor.id 기반 distinct user_id. org context는 별도 집계하지 않음.
total_events어떤 이벤트 타입을 포함하는지, 원천 row가 fact에서 어떻게 집계되는지event_count 합계. PushEvent, WatchEvent 등 type별 metric도 semantic layer에 정의.
retention_ratecohort 기준, week 0 포함 여부, 최신 미완성 기간 처리weekly lifecycle은 전주 active user 대비 기존 유저 비율. cohort heatmap은 first-seen week 기준.
trend_scorerecent/baseline window, 이벤트 가중치, seed affinity, 제외 키워드AI agent repo trend는 7일 recent, 직전 28일 baseline, weighted score와 seed affinity를 사용.
AI에게 "DAU SQL 짜줘"라고 하면 count(*)가 나올 수 있다. Analytics engineer는 count(distinct user_id)인지, bot을 제외할지, 날짜 timezone을 무엇으로 볼지, 최신 미완성 기간을 어떻게 표시할지까지 계약으로 만든다.

AI 시대의 Analytics Engineer

2025~2026년의 BI와 데이터 플랫폼은 자연어 질의, 데이터 agent, Copilot, semantic layer 쪽으로 빠르게 가고 있다. 그래서 AE의 역할은 줄어드는 게 아니라 더 선명해진다.

AI가 잘하는 일
  • 자연어 질문을 SQL 초안으로 바꾸기
  • 관련 테이블과 컬럼 후보 찾기
  • 기초 EDA, 차트, 요약문 생성
  • 이상치 후보와 세그먼트 후보 제안
  • dbt model/test 문서 초안 작성
AE가 책임져야 하는 일
  • 올바른 grain과 metric contract 승인
  • 중복, null, 기간, timezone, bot/자동화 계정 기준 검증
  • semantic layer에 지표를 등록하고 충돌 방지
  • dbt tests와 lineage로 자동 검증 가능하게 만들기
  • 운영 스케줄, 비용, 권한, 민감정보 경계를 관리
AI 분석 자동화의 병목은 SQL 생성이 아니라 신뢰 가능한 context다. agent가 안전하게 일하려면 fact grain, semantic metric, dbt test, lineage, 권한이 먼저 정리되어 있어야 한다.

AI가 데이터 분석을 자동화하는 프로세스

AI가 분석을 자동화한다는 말은 "모델이 아무 테이블이나 읽고 답한다"가 아니다. 안전한 흐름은 질문을 해석하고, 허용된 semantic layer에서 metric을 찾고, 검증 가능한 SQL과 결과를 만드는 쪽에 가깝다.

질문 해석"지난주 active user가 왜 떨어졌나"를 metric, 기간, 비교 기준으로 분해한다.
Context 검색dbt docs, semantic model, schema.yml, lineage에서 관련 fact/mart를 찾는다.
Grain 후보 선택일별, 주별, user-repo-action, cohort-week 중 질문에 맞는 해상도를 고른다.
SQL 생성metric 정의와 dimension을 기준으로 SQL을 만들고 BigQuery dry run으로 비용을 본다.
검증dbt tests, completeness, uniqueness, range, reconciliation 결과를 확인한다.
분석 확장세그먼트, 이벤트 믹스, cohort, repo별 기여도 등 follow-up 분석을 만든다.
해석 가드레일null은 미래 미관측인지, 상관과 인과가 섞였는지, 최신 기간이 완성됐는지 표시한다.
산출물 생성차트, 요약, notebook, dashboard, report 초안을 만든다.
제품화반복될 분석은 dbt model, semantic metric, test, Airflow DAG로 옮긴다.
사람 승인정의 변경, dashboard 반영, ML feature 사용은 사람이 검토하고 운영에 반영한다.

이 repo를 AI-ready하게 만드는 요소

dbt Semantic Layer

platform_activity.yml에 active_users, active_repos, total_events, event type별 metric을 선언한다. metric 이름과 집계 방식을 YAML로 고정해 BI/LLM/분석이 같은 정의를 쓰게 한다.

dbt tests

null/unique뿐 아니라 retention rate 범위, lifecycle accounting, heatmap active users와 WAU reconciliation을 검증한다. AI가 만든 답도 이 테스트를 통과한 mart 위에서만 신뢰할 수 있다.

Airflow 운영화

gharchive_dbt_metrics는 plan_fact_sync -> sync_fact -> build_dbt_metrics 순서로 실행된다. 분석 로직을 사람이 누르는 버튼이 아니라 반복 가능한 운영 파이프라인으로 바꾼다.

추천 feature mart

user_repo_interaction_mart, repo_feature_mart, experiment_split_mart는 window와 snapshot 기준을 둔다. AI/ML 실험에서 point-in-time 기준과 leakage 방지가 핵심이다.

AI agent trend mart

seed repo, keyword 후보, recent/baseline window, event weight, seed affinity, trend score를 명시해 agent 관련 repo trend를 재현 가능한 분석으로 만든다.

비용과 권한

BigQuery dry run, 90일 rolling window, full-history guardrail, 작은 metrics_* mart 조회가 있어야 AI가 질문을 많이 던져도 비용과 권한 경계를 관리할 수 있다.

AI 자동화까지 포함한 End-to-End 예시

DAU 감소 질문

  1. 사용자가 "지난주 DAU가 왜 줄었나?"라고 묻는다.
  2. agent는 semantic layer의 active_users와 activity_date를 찾는다.
  3. metrics_daily, metrics_event_type_daily에서 이벤트 믹스와 요일 효과를 비교한다.
  4. 최신 기간 completeness와 dbt test 통과 여부를 확인한다.
  5. 반복 질문이면 dashboard card와 alert 후보로 제품화한다.

Retention 해석 질문

  1. 사용자가 "이번 cohort retention이 낮은가?"라고 묻는다.
  2. agent는 first-seen cohort인지 lifecycle retention인지 먼저 확인해야 한다.
  3. 미완성 cohort의 null을 0으로 해석하지 않도록 가드레일을 붙인다.
  4. WAU reconciliation test와 rate range test 결과를 함께 보여준다.
  5. 해석이 확정되면 문서와 semantic metric 설명을 업데이트한다.

추천 모델 성능 질문

  1. 사용자가 "Two-Tower가 ALS보다 좋은가?"라고 묻는다.
  2. agent는 같은 experiment_split_mart와 같은 K 기준인지 확인한다.
  3. interaction grain, feature snapshot, seen filtering 기준을 검토한다.
  4. NDCG/Recall뿐 아니라 diversity와 cold-start 영향도 같이 본다.
  5. 성능 차이가 데이터 split 문제인지 모델 문제인지 분리한다.
