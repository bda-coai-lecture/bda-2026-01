왜 추천 시스템으로 보는가

비즈니스 관점

추천 시스템의 목적은 좋은 모델을 만드는 데서 끝나지 않는다. 사용자가 다음에 볼 만한 repo를 더 빨리 찾게 만들고, 탐색 시간을 줄이며, 서비스 안에서 더 많은 의미 있는 행동이 발생하게 만드는 것이 핵심이다.

  • 사용자 문제: GitHub에는 repo가 너무 많아서 관심 있는 프로젝트를 직접 찾기 어렵다.
  • 제품 기회: 개인화 추천, 유사 repo 추천, 트렌드 repo 추천은 탐색 비용을 낮춘다.
  • 성과 지표: 클릭률, 저장/Star, Fork, 재방문, 활동 지속률 같은 행동 지표로 검증해야 한다.
  • 리스크: 인기 repo만 반복 추천하면 다양성이 떨어지고, 작은 repo 발견 기회가 줄어든다.

개발 관점

개발의 핵심은 원천 로그를 모델 입력으로 바꾸고, 실험을 재현 가능하게 만들고, 학습 결과를 API가 안정적으로 읽을 수 있는 산출물로 고정하는 것이다.

  • 로그를 바로 모델에 넣지 않고, user-item interaction mart로 압축한다.
  • 시간 기준 split으로 train/test leakage를 막는다.
  • baseline, retrieval, ranker를 분리해 어떤 단계가 성능 병목인지 확인한다.
  • 모델 파일 단위가 아니라 bundle 단위로 서빙 버전을 관리한다.

개발 산출물

Raw log GitHub Archive event: actor, repo, event type, created_at 중심의 원천 로그
Interaction mart actor_id, repo_id 단위로 event weight를 적용한 feedback score 생성
Feature set 유저 활동량, repo 인기도, metadata, related candidate source, ALS score 등
Model artifacts ALS 후보, LGBM ranker, Two-Tower, FAISS index, 평가 summary
Serving bundle candidate cache, ranker, metric, manifest를 하나의 API 배포 단위로 관리

모델링 기준

항목 현재 기준
추천 구조 ALS retrieval → hybrid candidates → LGBM re-rank
대표 split history, rank-label, test를 시간 기준으로 분리
주요 지표 NDCG@100, Recall@100, unique recommended
성능 해석 Popularity < ALS < Two-Stage 흐름이 유지됨
남은 실험 negative sampling 재설계, neural ranker, source/feature ablation

단계별 상세 설명

1. 데이터 수집: 큰 로그를 비용과 재현 가능한 형태로 줄이기

첫 단계는 “추천 모델을 학습할 데이터가 있는가”가 아니라 “반복해서 다룰 수 있는 데이터 형태를 만들었는가”를 확인하는 단계다. GitHub Archive 원천 로그는 이벤트 단위라 크고, 같은 사용자가 같은 repo에 여러 번 행동하면 row가 계속 늘어난다.

관점 구체 설명
비즈니스 데이터 수집 비용이 너무 크면 실험을 자주 할 수 없다. 추천 품질 개선은 한 번의 학습보다 반복 실험이 중요하므로, BigQuery dry run으로 비용을 먼저 확인하고 필요한 컬럼과 기간만 추출한다.
개발 notebooks/gharchive/00_setup.ipynb에서 BigQuery 연결을 잡고, 01_extract_daily_agg.ipynb에서 일별 집계 Parquet를 만든다. 이후 모델 실험은 원천 로그를 매번 다시 쿼리하지 않고 data/daily_agg와 mart를 읽는다.
산출물 분석 가능한 Parquet 파일, 날짜 범위, row count, event type 분포, 이후 mart 생성을 위한 공통 입력 데이터.

2. 데이터 분석: 로그를 추천 문제로 번역하기

추천 모델은 “사용자와 아이템 사이의 관계”를 학습한다. 그래서 원천 event를 그대로 보는 대신, actor를 사용자, repo를 아이템, event를 관심 신호로 해석한다.

관점 구체 설명
비즈니스 모든 행동의 의미가 같지 않다. Watch는 관심 표현, Fork는 재사용 의도, Pull Request는 강한 기여 신호로 볼 수 있다. 반대로 Push는 본인 repo의 반복 작업일 수 있어 추천 신호로는 약하게 볼 수 있다.
개발 event type별 weight를 적용해 actor_id, repo_id 단위 score를 만든다. 이후 sparse matrix를 구성하고, repo metadata는 GitHub API 호출 비용과 rate limit을 줄이기 위해 SQLite cache로 관리한다.
검증 row count, null, 중복, 날짜 범위, event type 분포, 유저별 interaction 수, repo별 popularity long-tail을 확인한다. 이 단계에서 이상치를 놓치면 모델 성능 문제가 아니라 데이터 정의 문제가 된다.

3. 모델 실험: baseline부터 two-stage까지 단계적으로 개선하기

추천 모델 실험은 복잡한 모델을 바로 쓰는 방식이 아니라, 비교 기준을 만들고 하나씩 개선하는 방식으로 진행한다. 이 프로젝트에서는 Popularity, ALS, Two-Stage, Two-Tower를 순서대로 비교했다.

관점 구체 설명
비즈니스 모델 성능은 “정확도 하나”로 판단하지 않는다. 상위 10개 추천이 잘 맞는지, 100개 안에 정답 후보가 들어오는지, 매번 같은 인기 repo만 추천하지 않는지 함께 본다. 그래서 NDCG, Recall, coverage를 같이 본다.
개발 ALS는 전체 catalog에서 빠르게 후보를 뽑는 retrieval 역할을 맡고, LGBM은 그 후보를 다시 정렬하는 ranker 역할을 맡는다. 이 two-stage 구조는 실제 서비스에서도 자주 쓰이는 패턴이다. 전체 아이템을 매번 정렬하는 대신 후보 수백 개만 정교하게 정렬한다.
결과 현재 기준에서 Popularity보다 ALS가 낫고, ALS 후보를 LGBM으로 재정렬한 Two-Stage가 한 번 더 좋아진다. 최고 run은 Two-Stage/Fallback NDCG@100 = 0.025509, Recall@100 = 0.074161이다.
주의점 시간 기준 split을 지키지 않으면 test label이 학습에 섞이는 leakage가 생긴다. 최근 재설계 문서에서는 canonical positive, mixed negative sampling, ranker 학습 데이터 분리 문제를 따로 정리했다.

4. API 생성: 노트북 실험을 호출 가능한 제품 기능으로 바꾸기

모델 실험이 끝났다고 사용자가 추천을 받을 수 있는 것은 아니다. 학습 산출물을 API가 읽을 수 있게 묶고, 어떤 모델 버전이 현재 추천에 쓰이는지 명확히 관리해야 한다.

관점 구체 설명
비즈니스 서비스에서 중요한 것은 “제일 최근 모델”이 아니라 “검증되어 배포하기로 선택한 모델”이다. 성능이 좋은 후보라도 latency, coverage, fallback, 장애 대응이 준비되지 않으면 바로 적용하기 어렵다.
개발 src/ghrec/api.py를 FastAPI entrypoint로 두고, active bundle 조회, 추천 요청, 후보 source 확인, health check 같은 endpoint를 정의한다. bundle manifest에는 candidate cache, ranker model, metric summary, artifact path를 함께 저장한다.
계약 API 요청은 actor_id, k, bundle_id를 받고, 응답은 repo_id, rank, score, candidate_source, metadata를 돌려준다. unknown user, 후보 부족, fallback 같은 예외도 응답 정책에 포함해야 한다.

5. 서비스 적용·운영: 아직 남은 실제 제품화 단계

현재는 추천 모델 개발과 API 생성까지 온 상태다. 다음 단계는 추천을 실제 화면에 넣고, 사용자의 반응을 다시 데이터로 모아 모델과 제품을 계속 개선하는 운영 루프를 만드는 것이다.

관점 구체 설명
비즈니스 추천 위치를 정해야 한다. 개인화 홈에 넣을지, repo 상세 페이지의 유사 repo 영역에 넣을지, 신규 사용자를 위한 트렌드 추천으로 쓸지에 따라 성공 지표가 달라진다.
개발 프론트엔드는 API 응답을 카드 UI로 보여주고, 백엔드는 노출 로그와 클릭 로그를 남겨야 한다. 이후 offline metric과 online metric을 연결해 모델 개선이 실제 사용자 행동 개선으로 이어졌는지 본다.
운영 latency, error rate, empty result rate, 추천 coverage, 모델 drift, 데이터 freshness를 모니터링한다. 새 bundle은 자동 배포하지 않고 명시적 promotion과 rollback 정책을 둔다.

실험 결과를 해석하는 방식

비즈니스 해석

  • NDCG@100은 추천 후보 100개 안에서 정답이 얼마나 위쪽에 배치되는지 본다. 이 문서의 주 평가 기준이다.
  • Recall@100은 후보 100개 안에 실제 정답을 얼마나 놓치지 않고 포함했는지 본다. retrieval 병목 판단에 중요하다.
  • unique_recommended는 추천 다양성을 본다. 인기 repo만 반복 추천하는지 확인한다.
  • offline metric이 좋아도 서비스 KPI가 좋아진다는 보장은 없다. 그래서 적용 후 노출/클릭/행동 로그가 필요하다.

개발 해석

  • Popularity는 가장 단순한 기준선이다. 이것보다 못하면 개인화 모델을 쓸 이유가 약하다.
  • ALS는 collaborative signal로 후보를 잘 찾는지 본다. NDCG@100과 Recall@100으로 후보 품질을 비교한다.
  • LGBM re-rank는 ALS score, repo metadata, user activity, source feature를 함께 사용해 100개 후보의 순서를 다시 정한다.
  • FAISS는 모델 품질 개선이 아니라 검색 속도 개선이다. offline 모델을 실시간 API에 가깝게 연결하는 기술이다.

서비스 적용

추천 API를 제품 경험으로 연결

  • repo 상세 화면, 개인화 홈, 유사 repo 영역 중 어디에 추천을 둘지 결정한다.
  • 각 위치별 성공 지표를 다르게 잡는다. 예: 상세 화면은 유사 repo 클릭, 홈은 재방문과 탐색 깊이.
  • API 응답을 UI에 맞는 card/view model로 변환한다. repo name, language, stars, 설명, 추천 이유가 필요하다.
  • cold-start user, unknown actor, empty result 상황에서는 popularity 또는 recent fallback을 보여준다.
  • 추천 결과에 대한 노출 로그, 클릭 로그, 이후 행동 로그를 반드시 남긴다.
운영

모델을 계속 믿을 수 있게 관리

  • API latency, error rate, timeout, empty result rate를 기본 운영 지표로 본다.
  • 데이터 freshness와 feature 생성 실패를 감지한다. 오래된 데이터로 추천하면 품질이 빠르게 낡는다.
  • offline metric, online metric, 추천 coverage, item 다양성을 한 대시보드에서 본다.
  • 새 모델은 candidate 상태로 등록하고, metric과 샘플 결과 확인 후 promoted 상태로 올린다.
  • 문제가 생기면 이전 bundle로 rollback할 수 있어야 한다.
  • 주기적 재학습과 A/B test 정책을 정해 “한 번 만든 모델”이 아니라 “운영되는 시스템”으로 만든다.
