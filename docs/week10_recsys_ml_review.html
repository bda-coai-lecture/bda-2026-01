추천 문제 정의

추천이란, 다음 행동을 미리 맞추는 문제다

  • 기존 슬라이드 표현: 고객이 고를 것을 미리 맞추는 문제
  • GitHub에서는 고객이 actor_id, 제품이 repo_id다.
  • 정답은 클릭/구매 대신 Watch, Fork, PR, Issue, Star 같은 다음 행동이다.
  • 목표는 미래 test 기간에 interaction할 repo를 Top-K 안에 올리는 것이다.
Grain
user × repo

추천 모델은 raw event log를 직접 먹지 않는다. 먼저 유저-레포 단위의 Feedback Data로 압축한다.

추천은 "누가 무엇을 좋아할까"를 행렬 문제로 바꾸는 작업에서 시작한다.

비즈니스 목표

비즈니스 목표가 추천 문제를 정한다

  • 추천은 모델 점수를 높이는 일이 아니라 제품 행동을 바꾸는 일이다.
  • 목표가 discovery인지, 재방문인지, 구매/전환인지에 따라 정답 label이 달라진다.
  • 무엇을 추천하면 안 되는지도 중요하다. 품질, 정책, 재고, 신선도 제약이 들어간다.
  • 그래서 추천 시스템은 ML 문제이면서 동시에 제품/비즈니스 문제다.

GitHub repo 추천 예시

목표가 "새 repo 발견"이면 이미 본 repo는 제외한다. 목표가 "자주 쓰는 repo 빠른 재접근"이면 이미 본 repo를 다시 보여주는 것도 맞을 수 있다.

같은 데이터라도 비즈니스 목표가 바뀌면 filtering, label, metric, fallback이 같이 바뀐다.

Sparsity

대부분의 칸은 비어 있다

  • 가능한 user-repo 조합은 너무 많다.
  • 실제로 관측된 interaction은 극히 일부다.
  • 빈 칸은 "싫다"가 아니라 "모른다"에 가깝다.
Mental model
mostly empty

유저 x 아이템 후보 공간은 폭발적으로 커지지만 실제 관측은 극히 일부다. 추천은 이 빈칸을 채우는 문제다.

Sparse matrix

추천에서는 Sparse matrix가 기본이다

기존 표현 그대로: 추천에서는 dense table이 아니라 sparse matrix가 일반적이다. 비어 있는 칸을 전부 저장하지 않는다는 뜻으로 이해하면 충분하다.

최종 복습의 기준

데이터를 나누는 이유는 미래를 숨기기 위해서다

History
모델이 볼 수 있는 과거유저 활동, repo 활동, feature를 만드는 기준 기간
Input
Future
모델이 맞혀야 하는 미래학습/평가에서 정답 확인에 쓰는 다음 행동
Label
Rule
미래 행동은 feature에 넣지 않는다미래를 섞으면 추천 모델이 아니라 답안지를 본 모델이 된다.
No leak
Two-stage의 두 모델이 서로 다른 데이터셋을 쓴다는 뜻이 아니다. 같은 추천 데이터 위에서 후보 생성과 정렬이 이어지고, 미래 행동은 정답 확인용으로만 분리한다.

Seen filtering

이미 본 item을 제외할지는 서비스마다 다르다

  • 지금 GitHub repo 추천 실험에서는 이미 본 repo를 기본적으로 제외한다.
  • 새 repo 발견이 목표라면 반복 추천은 피로도를 만들 수 있다.
  • 반대로 음악, 영상, 장보기처럼 재소비가 자연스러운 서비스는 다시 추천할 수 있다.

정책의 문제

seen filtering은 모델 알고리즘이 아니라 제품 정책이다. 실험 결과를 볼 때는 "무엇을 추천 가능한 item으로 봤는지"를 먼저 확인해야 한다.

이번 실험의 정책: 이미 interaction한 repo는 제외하고, 아직 볼 가능성이 있는 repo를 찾는다.

Baseline & Fallback

인기순/최신순은 강한 baseline이자 fallback이다

왜 필요한가

  • 개인화 추천의 개선을 증명하는 기준선이다.
  • Cold Start와 장애 상황에서 fallback이 된다.
  • 추천 서버가 실패해도 빈 화면을 피할 수 있다.

무엇이 부족한가

  • 개인화가 없다.
  • 추천 다양성이 낮다.
  • 인기 repo만 반복 노출한다.

Part 2

Matrix Factorization

큰 유저-아이템 행렬 R을 작은 두 행렬 P와 Q로 근사한다. 유저와 아이템을 같은 숨은 취향 공간에 놓는다는 뜻이다.

MF intuition

R ≈ P × Q, 숨은 취향 차원을 찾는다

  • user vector는 사용자의 잠재 취향이다.
  • item vector는 repo의 잠재 특성이다.
  • 두 벡터의 내적이 높으면 추천 점수가 높다.
Score
u · i

기존 표현: 예측 = 유저 벡터 · 아이템 벡터. 방향이 비슷할수록 내적값이 커지고 높은 예측 점수가 된다.

ALS / BPR

ALS는 확신도를 두고, BPR은 순서를 맞춘다

ALS
P 고정 → Q 최적화, Q 고정 → P 최적화많이 본 아이템일수록 "선호한다고 더 믿는다"는 가중치를 둔다.
Score
BPR
본 것 > 안 본 것정확한 점수보다 본 repo가 안 본 repo보다 앞서도록 학습한다.
Order
공통점
둘 다 embedding을 얻는다학습된 user/item 벡터는 연관 추천과 빠른 검색에 활용된다.

ALS limits

하지만 ALS만으로는 부족하다

  • 신규 유저와 신규 아이템에는 약하다.
  • language, topic, stars 같은 side feature를 직접 쓰기 어렵다.
  • 최종 제품에는 다양성, 신선도, 정책 제약도 필요하다.

해석

ALS는 강한 기준선이지만 완성된 추천 제품은 아니다. retrieval source 중 하나로 보는 편이 안전하다.

Embedding

Embedding은 상품/유저에 대한 숫자의 집합이다

가까운 벡터

비슷한 repo, 비슷한 user로 해석할 수 있다.

내적 점수

방향이 비슷할수록 추천 점수가 커진다.

검색 문제

Top-K 추천은 nearest neighbor search가 된다.

Repo2Repo

item factor끼리 비교하면 related repo 후보를 만들 수 있다.

BPR 관점

절대 점수보다 본 item이 안 본 item보다 위에 오게 학습한다.

서빙 연결

embedding은 빠른 검색과 미리 계산한 후보 목록으로 이어진다.

추천 모델의 output은 단순 점수표가 아니라 검색 가능한 벡터 공간이다.

Part 3

Embedding Search

벡터가 생기면 추천은 가까운 이웃 찾기가 된다. 정확하게 전부 비교하면 느리고, 그래서 빠른 검색 도구를 쓴다.

빠른 벡터 검색

FAISS는 retrieval layer의 속도 장치다

방식역할강의 메시지
FlatIP모든 벡터를 내적 점수로 빠르게 비교5강 벤치마크 기준 2.4ms, recall 100%라 기준으로 쓰기 좋다.
더 빠르게 찾기일부 후보만 먼저 좁혀서 검색catalog가 훨씬 커졌을 때 쓰지만, 너무 줄이면 정답을 놓칠 수 있다.
온라인 흐름유저 벡터 -> FAISS -> 후보 repo모델 품질 개선이 아니라 서빙 가능한 retrieval로 바꾸는 장치다.
이번 복습에서는 "정확하게 찾는데도 충분히 빠른 방식"을 기준으로 잡는다. 더 어려운 인덱스 튜닝은 catalog가 훨씬 커졌을 때의 주제다.

Two-Tower

Two-Tower는 유저와 아이템을 따로 인코딩한다

  • User tower: 유저 id, 활동 요약, 선호 신호
  • Item tower: repo id, language, stars, forks, metadata
  • 두 tower output을 같은 embedding 공간으로 맞춘다.

왜 두 개로 나누나?

미리
모든 repo를 벡터로 저장repo는 많지만 자주 바뀌지 않으므로 미리 계산해 둔다.
요청
현재 유저를 벡터로 변환유저의 활동 요약으로 "이 유저가 좋아할 방향"을 만든다.
검색
가까운 repo 벡터를 후보로 선택그 다음 ranker가 후보의 최종 순서를 정한다.

운영 해석

Two-Tower는 실패가 아니라 역할이 달랐다

  • 정확도만 보면 ALS가 우세하다.
  • Unique@100을 보면 Two-Tower가 catalog를 넓게 연다.
  • 따라서 "대체"보다 "candidate source 추가"가 맞다.
Recipe

Part 4

Two-stage Architecture

전체 아이템을 한 번에 정렬하지 않는다. Stage 1은 빠르게 후보를 넓게 찾고, Stage 2는 후보를 정교하게 다시 정렬한다.

Candidate Generation + Re-ranking

Retrieval은 놓치지 않는 일, Ranking은 위로 올리는 일

Stage 1
Candidate Generation목표는 precision보다 recall이다. 후보 안에 정답이 들어오게 만든다.
Candidate
Stage 2
Re-rankingLGBM이 feature를 보고 후보의 순서를 고친다.
Top-K
두 stage는 같은 학습 기간 데이터와 같은 추론 시점 데이터 위에서 앞뒤로 이어진다. 다만 역할이 후보 찾기와 순서 고치기로 나뉜다.

Candidate source를 섞는다

후보 소스는 하나일 필요가 없다

  • ALS head: 정확도 높은 collaborative 후보
  • ALS tail: 덜 뻔한 후보 보강
  • Recent/popular: fallback과 trend 대응
  • Related repo: repo-to-repo 탐색
  • Two-Tower: side feature 기반 diversity

Ranker가 보는 것

후보 source 자체도 중요한 feature다. 어떤 경로로 들어온 후보인지가 최종 정렬에 도움이 된다.

피처 조합

Re-ranking은 피처 조합을 보는 문제다

User feature

활동량, 최근성, 선호 language/topic

Item feature

stars, forks, language, freshness, popularity

Pair feature

ALS score, repo similarity, seen distance

Source feature

ALS, popular, recent, related, two-tower

Context feature

기간, catalog, candidate rank

Label

미래 기간 interaction 여부

기존 슬라이드 예시처럼 시간=12시와 선호=한식은 따로 보면 약하지만, 조합하면 "점심에 한식 도시락" 같은 강한 추천 신호가 된다.

Deep ranker 해석

DeepFM, DLRM이 항상 이기지는 않는다

  • positive label이 적으면 neural ranker가 안정적으로 학습하기 어렵다.
  • tabular feature가 강하면 tree model이 매우 강한 baseline이다.
  • 딥러닝은 데이터량, negative sampling, calibration까지 같이 봐야 한다.

강의 메시지

"복잡한 모델 = 더 좋은 모델"이 아니다. 같은 split에서 단순한 강한 baseline을 이겨야 한다.

Recall@K좋은 후보를 충분히 놓치지 않고 모았는가?candidate source 추가, fallback 후보 보강
Unique@K모두에게 같은 인기 item만 보여주고 있지 않은가?롱테일 노출, 다양성 슬롯, 신규 item 탐색
Latency제품 경험을 해치지 않는 시간 안에 응답하는가?precompute 범위, FAISS 선택, API timeout 정책
강의에서 보는 offline score는 출시 결정의 전부가 아니다. 실제로는 KPI, UX, 정책 리스크, 운영 비용까지 같이 보고 실험한다.

온라인 실험

출시는 A/B test와 guardrail로 판단한다

Primary KPI

클릭, star/fork/watch, 재방문, 전환처럼 이번 추천이 직접 바꾸려는 행동

Guardrail

응답 지연, 오류율, 신고/숨김, 세션 이탈처럼 악화되면 안 되는 지표

Segment

warm user, cold user, heavy user, 신규 repo 노출처럼 효과가 다르게 나오는 그룹

Exposure

추천이 실제로 보였는지, 어느 위치에 보였는지, 사용자가 클릭 가능했는지

Policy

중복, 이미 본 item, 민감/저품질 item, 비즈니스 우선순위 제약

Rollback

모델 버전, fallback, 이전 추천 bundle로 되돌아갈 수 있는 운영 절차

좋은 offline 모델도 제품 실험에서 이기지 못할 수 있다. 그래서 추천은 모델 평가와 제품 실험을 함께 설계한다.

Leakage

성능 수치보다 먼저 leakage를 본다

  • ranker train label과 eval label이 같은 기간에서 오면 안 된다.
  • 미래 popularity나 최신 metadata를 과거 예측 feature로 쓰면 위험하다.
  • notebook 실험과 production script 결과를 구분해야 한다.

Part 6

Cold Start와 Fallback

기존 5강 표현 그대로, Cold Start는 학습할 데이터가 없는 신규 유저/아이템 문제다. 그래서 Baseline & Fallback이 필요하다.

Cold Start

신규 유저/아이템은 행렬에 행이나 열이 없다

  • cold user는 history가 없어 ALS 개인화 후보를 만들기 어렵다.
  • cold item은 interaction이 없어 collaborative 모델에 잘 들어오지 않는다.
  • fallback은 예외 처리가 아니라 제품 정책이다.

현재 구현 상태

미리 계산한 후보 목록에 actor가 없으면 API는 ActorNotFoundError를 내고, simulator가 Cold Start 안내를 보여준다. 즉 fallback 결과를 API가 직접 반환하는 단계는 아직 아니다.

10주차 과제: unknown actor에게 무엇을 보여줄지 명시적으로 설계하기.

Fallback 설계

Warm user와 Cold Start를 나눠 본다

상황 쓸 수 있는 신호 추천 정책 강의 메시지
Warm user history repo, weighted score, user profile ALS / Two-Tower 후보 + LGBM re-rank 개인화 모델의 주 무대
User Cold Start 없거나 매우 적은 history recent popular, trending, onboarding language/topic fallback 품질이 첫 경험을 좌우
Item Cold Start language, topics, owner, stars, text metadata same topic/language, related repo, exploration slot side feature와 탐색 정책이 필요

현재 vs 운영

Cold Start 응답은 404에서 끝나면 안 된다

현재
unknown actor → 404앱이 빈 결과와 안내 문구를 보여준다.
설명용
Production
unknown actor → fallback Top-K인기순, 최신순, 트렌디 repo, onboarding topic을 섞고 cold_start=true를 내려준다.
제품용

모니터링해야 할 값

  • cold_start_rate
  • empty_result_rate
  • fallback_source_mix
  • cold/warm segment quality

Item Cold Start

신규 repo는 collaborative 모델에 바로 잡히지 않는다

  • interaction이 없으므로 ALS factor가 약하거나 없다.
  • language, topic, owner, README, stars 같은 metadata가 필요하다.
  • exploration slot으로 신규 repo 노출 기회를 만든다.

해결 방향

content-based retrieval, repo-to-repo related graph, topic/language fallback이 cold item 문제의 출발점이다.

체크리스트

추천 실험을 볼 때 반드시 묻는 질문

같은 split인가?

history, rank_label, test 기간이 다르면 수치를 직접 비교할 수 없다.

같은 catalog인가?

전체 repo인지 100k catalog인지에 따라 recall과 diversity가 바뀐다.

비즈니스 목표가 무엇인가?

discovery, 재방문, 전환 중 무엇을 최적화하는지에 따라 label과 metric이 달라진다.

seen filtering이 일관적인가?

이미 본 repo를 추천하면 평가와 제품 경험이 모두 왜곡된다.

cold-start를 어떻게 처리하는가?

unknown actor와 신규 repo는 별도 fallback 정책이 필요하다.

metadata leakage는 없는가?

stars/forks/topics가 현재 snapshot이면 과거 예측 평가에서는 조심해야 한다.

serving까지 연결되는가?

offline 점수, 빠른 검색 구조, API 응답, UI fallback이 한 흐름이어야 한다.

Part 7

AI 시대의 추천시스템

마지막으로, 오늘 배운 추천 기본기가 LLM/RAG 시대에는 어떻게 이어지는지 정리한다. 결론은 대체가 아니라 연결이다.

AI 시대의 추천

Vector search가 추천과 RAG를 다시 연결했다

  • RAG 시스템은 문서 chunk를 embedding으로 바꾸고 가까운 것을 찾는다.
  • 추천에서도 user/item embedding을 만들고 가까운 item을 찾는다.
  • 그래서 FAISS, nearest neighbor, candidate generation은 AI 앱에서도 계속 등장한다.
  • 다만 vector search만으로 제품 품질이 자동으로 해결되지는 않는다.

같은 구조

RAG: query vector → document vector 검색

추천: user vector → item vector 검색

둘 다 retrieval layer를 가진다. 그 다음에는 filtering, ranking, policy, latency가 따라온다.

현재 해석

급격한 대체보다, 검증된 기본기 위에 AI를 얹는 흐름이다

  • 최근 RAG도 vector-only보다 keyword, metadata, plain text 정제, index map을 함께 본다.
  • LLM 붐 속에서 대규모 추천 자체가 전면의 키워드로 보이는 분위기는 아직 강하지 않다.
  • 현실적인 변화는 비정형 데이터를 추천 feature로 가공하기 쉬워진 점이다.
  • 또 하나의 변화는 API 개발, 실험 앱, 운영 자동화의 진입장벽이 낮아진 점이다.

강의 결론

추천시스템을 배운다는 것은 낡은 기술을 배우는 게 아니다. sparse data, retrieval, ranking, fallback, API contract는 AI 제품에서도 그대로 필요한 기본기다.

LLM은 추천을 없애기보다, 텍스트 이해와 시스템 제작 속도를 높이는 도구로 먼저 들어온다.
